# ADR-003: Expand V1 Scope to Include Locale URL Routing and Language Switcher

**Date:** 2026-05-07
**Status:** Accepted

## Context

The original V1 scope (ADR-001) deferred locale-aware URL routing and a language switcher to a future task, delivering only the data foundation (folder structure, DB schema, conventions). During PRD review, the author clarified that adding Portuguese posts without locale routing would result in bilingual content being served on the same URL with no way for readers to choose a language. This makes the "multilingual" feature non-functional from a reader perspective.

The question became: ship data-only now and add routing later, or include routing in this task to make bilingual truly functional end-to-end.

## Decision

Expand V1 scope to include:
1. Locale-aware URL routing: `/en/<slug>` and `/pt-br/<slug>` for posts, `/en/blog` and `/pt-br/blog` for listings
2. A persistent language switcher in the header that links to the same post in the alternate locale (falls back to the locale listing if no translation exists)
3. `hreflang` meta tags on post pages linking the two locale versions
4. The old `/blog` and `/<slug>` routes redirect to `/en/blog` and `/en/<slug>` respectively (English as default)

Listing model: separate per-locale pages (`/en/blog`, `/pt-br/blog`), each showing only posts in that language. No mixed feed.

## Alternatives Considered

### Alternative 1: Data-only V1 (original ADR-001 scope)

- **Description:** Ship folder structure + DB schema only. Add routing in a separate task.
- **Pros:** Smaller PR; clear separation of concerns.
- **Cons:** After merge, Portuguese posts can be added to the DB but can't be served to readers in a usable way. The blog appears bilingual in the DB but monolingual to readers. The routing task becomes a blocker before the first Portuguese post can actually go live.
- **Why rejected:** The author's goal is to write in both languages. A data foundation without serving capability delays that goal by a full task cycle.

### Alternative 2: Single mixed listing with language filter

- **Description:** `/blog` shows all posts; a language filter lets readers toggle between `en` and `pt-br`.
- **Pros:** Single listing URL, simpler routing.
- **Cons:** Mixed-language feed is a documented UX anti-pattern (readers are confused when a Portuguese post follows an English post with no clear separation). Requires more complex filtering logic for an uncertain UX benefit.
- **Why rejected:** Separate listing pages per locale is the industry standard (market research confirmed); simpler reader mental model.

## Consequences

### Positive

- Bilingual support is complete and functional after one task — the author can write in Portuguese and readers can access it immediately
- `hreflang` tags are meaningful from the start (both locale URLs exist)
- Language switcher provides immediate cross-language navigation
- English as default locale preserves backward compatibility for existing readers and links

### Negative

- Larger PR (~12-15 files vs. ~8) — acceptable for a solo developer project
- TanStack Router locale layout requires careful route nesting to avoid breaking existing routes
- Redirect from `/blog` → `/en/blog` and `/<slug>` → `/en/<slug>` must be implemented to preserve any existing links

### Risks

- TanStack Router i18n layout complexity may surface unexpected routing edge cases — mitigation: run `make test` and manual dev server testing before merge
