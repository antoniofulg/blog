# ADR-004: Technical Architecture ‚Äî i18n Routing, Locale Preference, and Translation Fallback

**Date:** 2026-05-07
**Status:** Accepted

## Context

Three technical decisions needed resolution before writing the TechSpec:

1. **Routing shape:** How to implement `/en/$slug` and `/pt-br/$slug` in TanStack Router's file-based system without duplicating route logic.
2. **Locale persistence:** Where to store the user's language preference (like the theme toggle).
3. **Missing translation behavior:** What to render when a user navigates to `/pt-br/react-suspense` and no Portuguese version exists.

## Decision

### Routing: Layout Route with `$lang` Parameter

Use a single layout file `app/routes/$lang.tsx` with a `$lang/` directory containing child routes (`$lang/blog.tsx`, `$lang/$slug.tsx`). TanStack Router nests these automatically. The layout route validates `$lang` in its `beforeLoad` ‚Äî throws a redirect to `/en/blog` if the param is not `'en'` or `'pt-br'`.

Old routes (`/blog`, `/$slug`) keep their files but their loaders redirect to the locale-equivalent URL, using the stored locale preference from localStorage (default `'en'`).

### Locale Persistence: localStorage, Mirroring the Theme System

Create `app/lib/locale.tsx` following the exact same pattern as `app/lib/theme.tsx`: a `LocaleProvider` wrapping the app in `__root.tsx`, a `useLocale()` hook, and `localStorage` key `'locale'` with default `'en'`. The language switcher button in the header reads `useLocale()`, writes the new locale to localStorage, and navigates to the locale-prefixed URL for the current page.

### Translation Fallback: Serve English with Notice

When a user navigates to `/pt-br/<slug>`:
1. The loader queries `(slug, 'pt-br')` in the DB.
2. If not found, queries `(slug, 'en')`.
3. If English found: renders the English post with a `"notTranslated"` flag; the UI shows a "this post is not available in Portuguese yet" notice.
4. If neither found: throws `notFound()` ‚Üí 404 page.

## Alternatives Considered

### Routing Alternative: Flat Duplicate Files per Locale

- `app/routes/en/$slug.tsx`, `app/routes/pt-br/$slug.tsx`, etc.
- **Rejected:** Every route file duplicated. Adding a third locale means duplicating all files again. Violates DRY.

### Locale Persistence Alternative: Cookie-Based

- Store locale in a cookie; derive it server-side in `beforeLoad`.
- **Rejected:** Adds SSR complexity. The theme system uses localStorage successfully with `suppressHydrationWarning`; locale follows the same pattern with less overhead.

### Translation Fallback Alternative: Disable Switcher for Untranslated Posts

- Grey out the language switcher when no translation exists.
- **Rejected:** Poor UX ‚Äî users can't explore the Portuguese catalog from an untranslated English post. The English-with-notice approach is more informative.

### Translation Fallback Alternative: Redirect to Locale Listing

- `/pt-br/react-suspense` ‚Üí redirect to `/pt-br/blog` when no translation.
- **Rejected:** User loses context of what they were reading. English fallback with notice is more respectful of user intent.

## Consequences

### Positive

- One layout file handles all locale validation ‚Äî adding a third locale (e.g., `es`) requires only updating the `LOCALES` constant
- Locale persistence mirrors the theme pattern exactly ‚Äî consistent mental model for the codebase
- Fallback rendering means no dead ends for users on untranslated posts

### Negative

- `$lang/$slug.tsx` loader makes two DB queries in the fallback case ‚Äî acceptable for a low-traffic personal blog
- Old routes (`/blog`, `/$slug`) need client-side localStorage read to determine the redirect target ‚Äî adds a brief render before redirect; mitigated by `suppressHydrationWarning` already on `<body>`
- TanStack Router `$lang` parameter captures all single-segment paths ‚Äî must validate to avoid `/$lang` matching unrelated paths (mitigated by `beforeLoad` redirect on invalid values)
