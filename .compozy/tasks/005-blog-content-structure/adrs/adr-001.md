# ADR-001: V1 Scope — Conventions + DB Schema, No UI Filtering

## Status

Accepted

## Date

2026-05-07

## Context

The blog has 3 MDX posts in a flat `content/` directory with no category, series, or image organization. As post count grows, there is no structural way to group content, find related posts, or navigate by topic. The question is how much infrastructure to build now versus later.

Two camps emerged in the council debate: build the full DB schema now while the migration is cheap, or defer DB columns until 10+ posts provide real evidence of what categories and series patterns are needed.

## Decision

Add `category` (text, nullable), `series` (text, nullable), and `seriesPart` (integer, nullable) columns to the `posts` DB table and update the indexer to read these fields from MDX frontmatter. Use plain text for `category` — no database enum — so the vocabulary can evolve without a migration. Define image folder and branch naming conventions as documented rules with no code enforcement in V1. Do not build UI category filtering or series navigation in V1.

## Alternatives Considered

### Alternative 1: Docs-only (no DB changes)

- **Description**: Write a CONVENTIONS.md file. No schema changes. Add DB columns when posts > 10.
- **Pros**: Zero risk of over-engineering a 3-post blog; zero migration needed.
- **Cons**: Conventions without enforcement drift. Retroactive migration is harder than proactive migration.
- **Why rejected**: The DB migration cost is near-zero now. Adding columns to an empty table is trivial. Deferring adds no real benefit and creates future friction.

### Alternative 2: Full content platform (V1 with UI filtering)

- **Description**: DB schema + indexer + category filter UI on blog listing + series navigation component.
- **Pros**: Complete content discovery for readers from day one.
- **Cons**: No posts to filter. Premature UI for a blog with < 10 posts. High implementation cost for zero reader benefit.
- **Why rejected**: Council consensus: no UI until posts > 10 and navigation is a real user need.

## Consequences

### Positive

- `category` and `series` fields available in DB immediately — no retroactive migration when posts grow
- Plain-text `category` allows vocabulary evolution without schema changes
- Image folder and branch naming conventions documented before the blog scales
- Indexer updated once; all future posts automatically indexed with new fields

### Negative

- Nullable columns add minor schema complexity
- Category vocabulary is unconstrained in V1; post authors can introduce typos (mitigated by documented controlled list)

### Risks

- **Taxonomy rot**: If the 6-category list proves wrong, frontmatter edits are needed across all posts. Mitigation: keep list small (6 categories), use plain text not enum, and treat the vocabulary as soft until posts > 10.
- **Series field unused**: If series pattern never emerges, `series` and `seriesPart` columns are dead weight. Mitigation: nullable with zero performance cost; easy to drop later if never used.

## Implementation Notes

- Add migration file via `drizzle-kit generate` after schema update
- Update `parseFrontmatterBlock` in `indexer.ts` to read `category`, `series`, `seriesPart`
- Image folder convention: `public/images/<slug>/` — no code change, documented pattern only
- Branch naming for posts: `post/<slug>` — git convention, enforced by documentation
