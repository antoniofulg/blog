# ADR-005: Initial Locale Detection — Accept-Language Header + Portuguese Country Mapping

**Date:** 2026-05-07
**Status:** Accepted

## Context

When a user visits the site at a locale-less URL (`/`, `/blog`, `/$slug`) with no stored locale preference in localStorage, the app must pick a starting language. The user specified that the site should "get the user location to set the language" and map Portuguese-speaking locations to `pt-br`, defaulting to English otherwise.

Two signal sources are available without external service dependencies:
1. `Accept-Language` HTTP header — set by the browser, reflects the user's OS/browser language preference
2. IP geolocation — requires an external API (e.g., `ipapi.co`) or a bundled GeoIP database

## Decision

Use the `Accept-Language` request header as the primary locale signal. Parse it server-side in the `beforeLoad` of locale-less entry routes. If any `pt-*` locale appears in the header (e.g., `pt-BR`, `pt-PT`, `pt`), redirect to `pt-br`. Otherwise, redirect to `en`.

Detection priority:
1. **localStorage `'locale'` key** — explicit user choice, always wins; checked client-side before server redirect fires
2. **`Accept-Language` header** — parsed server-side in `beforeLoad`; mapped: `pt-*` → `pt-br`, everything else → `en`
3. **Default: `en`** — when header is absent or unreadable

The Portuguese locale set covers: `pt`, `pt-BR`, `pt-PT`, `pt-AO`, `pt-MZ`, `pt-CV`, `pt-GW`, `pt-ST`, `pt-TL`. Any header value starting with `pt` (case-insensitive) maps to `pt-br`.

## Alternatives Considered

### IP Geolocation API (e.g., `ipapi.co`)

- **Description:** On first visit, call an external API with the client IP to get the user's country code; map known Portuguese-speaking countries to `pt-br`.
- **Pros:** Country-based — a Brazilian using English-language Chrome still gets `pt-br`; more accurately reflects the user's location rather than browser language setting.
- **Cons:** External API dependency (rate limits, downtime, latency); privacy concern (IP sent to third party); adds an async network call to every cold-first visit; needs API key management.
- **Why rejected:** External dependency on a free-tier API is fragile for a personal blog. `Accept-Language` covers the primary use case (Brazilian users typically have `pt-BR` in their browser language settings) without the cost.

### Bundled GeoIP Database (MaxMind GeoLite2)

- **Description:** Bundle a local GeoIP database, query it with the client IP to get country code.
- **Pros:** No external API call; country-level accuracy.
- **Cons:** Binary database file adds ~30MB to the repo; requires periodic updates to stay accurate; setup complexity disproportionate to a personal blog's needs.
- **Why rejected:** Over-engineered for a personal blog. Accept-Language is sufficient.

### Browser Geolocation API

- **Description:** Call `navigator.geolocation.getCurrentPosition()` on mount; map coordinates to country.
- **Pros:** Most accurate physical location.
- **Cons:** Requires browser permission prompt (terrible UX for a blog); async; client-side only (flash of wrong language before detection).
- **Why rejected:** Permission prompt on a blog visit is a poor UX pattern.

## Consequences

### Positive

- Zero external API dependency — works offline, no rate limits, no API key
- Detection happens server-side in `beforeLoad` — no client-side flash
- Brazilian users using Portuguese-language browsers get `pt-br` automatically
- User's explicit choice (localStorage) always overrides the detection

### Negative

- A Brazilian developer using English-language Chrome gets `en` on first visit — they must manually switch once (then localStorage persists their choice)
- No country-level detection in V1 — Portuguese speakers with English-language browsers need one manual switch
