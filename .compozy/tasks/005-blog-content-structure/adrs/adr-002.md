# ADR-002: Atomic Single-PR Delivery Strategy

**Date:** 2026-05-07
**Status:** Accepted

## Context

The content structure task touches three distinct layers: file organization (moving posts to locale subdirectories), database schema (new columns + unique constraint change), and developer conventions (CONTENT.md, branch naming, CI lint). The question is whether to deliver these as one atomic PR or split into separate PRs for reviewability.

## Decision

Deliver all changes in a single atomic PR on branch `TASK-0005/blog-content-structure`:
- Move existing 3 posts from `content/` to `content/en/`
- Create `content/pt-br/` directory scaffold
- DB migration: add `lang`, `category`, `series`, `seriesPart`, `draft` columns; change `slug UNIQUE` to `UNIQUE(slug, lang)`
- Update indexer to derive `lang` from file path and read new frontmatter fields
- Write `CONTENT.md` conventions document
- Add `post/<lang>/<slug>` to git-workflow rules
- Add frontmatter CI lint test with blocking category enforcement

## Alternatives Considered

### Alternative 1: Split PRs (conventions first, schema second)

- **Description:** PR 1 moves files and documents conventions; PR 2 adds DB columns.
- **Pros:** Smaller PRs, easier to review individually.
- **Cons:** After PR 1 merges, `content/en/` exists but the DB doesn't know about `lang` — the indexer either silently ignores locale or produces incorrect data. Interim broken state.
- **Why rejected:** The file-move and schema-change are tightly coupled. An interim state where locale subdirectories exist but the DB has no `lang` column creates inconsistency that could corrupt the posts table.

### Alternative 2: Atomic + hreflang

- **Description:** Everything from Approach A plus `hreflang` meta tags in post route head.
- **Pros:** SEO benefit from launch.
- **Cons:** `hreflang` requires both locale URLs to exist to be valid. No pt-BR posts exist yet, so the tags would reference non-existent URLs — worse for SEO than no tags at all.
- **Why rejected:** Premature SEO optimization; deferred to V2 when the first translation pair ships.

## Consequences

### Positive

- Repo is in a fully consistent state after one merge — no interim period with mismatched file structure and schema
- DB migration runs against a nearly-empty table — trivial cost
- Conventions and schema align from day one; all future posts follow the same structure

### Negative

- Single PR touches ~8 files — larger diff to self-review
- Accepted for a solo developer project; benefit of atomic consistency outweighs review overhead
