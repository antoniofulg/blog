# ADR-006: Modal primitive = @radix-ui/react-dialog

## Status

Accepted

## Date

2026-05-21

## Context

PRD F7 requires a reusable accessible dialog primitive in `app/components/ui/`, consumed by F4 (missing-twin confirm modal). The TechSpec codebase scan confirmed the project has no existing dialog/modal primitive and no `@radix-ui/*`, shadcn, or headless-ui dependencies. Three approaches were considered:

1. **Native `<dialog>` + Tailwind + small hook** — zero new deps; HTML5 `<dialog>` handles focus management and Escape natively; Chrome 37+, Safari 15.4+, Firefox 98+ — all evergreen for the audience.
2. **Add `@radix-ui/react-dialog`** — battle-tested, full a11y semantics, portal-based, ~12 kB gzipped, opens the door to a future shadcn-style component set.
3. **Custom React portal + focus-trap-react** — maximum control, most maintenance, equivalent dep cost.

User selected option 2 during TechSpec clarification.

## Decision

Add `@radix-ui/react-dialog` as a runtime dependency. Build `app/components/ui/dialog.tsx` as a thin wrapper exposing a `<Dialog>` component composed of Radix primitives (`Root`, `Trigger`, `Portal`, `Overlay`, `Content`, `Title`, `Description`, `Close`), styled with Tailwind utility classes. The wrapper follows the shadcn-style compositional convention (named subcomponents on a default export) so future dialog use sites can reuse it without re-binding Radix internals.

The confirm modal for the missing-twin flow is built as `app/components/ui/missing-twin-dialog.tsx`, consuming `<Dialog>` and rendering the current-page-language copy from an inline `Record<Locale, { title: string; body: string; confirm: string; cancel: string }>` (matches the existing inline-string convention in `language-menu.tsx:6-24`).

## Alternatives Considered

### Alternative 1: Native `<dialog>` element + Tailwind + custom hook

- **Description**: Use the HTML5 `<dialog>` element with `.showModal()` / `.close()`. Tailwind for styling. A ~30-line `useDialog` hook handles SSR-safe ref guarding and the open/close API.
- **Pros**: Zero new deps; browser handles focus trap, Escape, return-focus-on-close; smallest bundle impact.
- **Cons**: Cross-browser quirks remain (Safari `::backdrop` styling, animation interruption); no animation primitives baked in; future use sites need to re-bind the hook each time; less batteries-included for advanced use cases (nested dialogs, multi-step modals).
- **Why rejected**: The 12 kB cost of Radix is acceptable; getting battle-tested a11y + portal behavior out of the box is worth more than dep avoidance for a primitive that is likely to be reused (e.g., confirm dialogs for future destructive admin actions, share modals).

### Alternative 2: Custom React portal + focus-trap-react

- **Description**: Roll a custom `<Dialog>` with `createPortal` and `focus-trap-react` for a11y.
- **Pros**: Maximum implementation control; smaller than Radix.
- **Cons**: Equivalent dep cost (`focus-trap-react` ≈ 5 kB) without the ecosystem benefits; reinvents wheel; long-term maintenance burden.
- **Why rejected**: No advantage over Radix at comparable size, and worse long-term value.

## Consequences

### Positive

- Production-grade a11y semantics (focus trap, ARIA roles, Escape, return-focus, portal stacking) without reinventing.
- Wrapper API matches the shadcn ecosystem — future imports (e.g., adding `<Tooltip>`, `<Popover>`) can follow the same pattern without churn.
- `<Dialog>` becomes the primitive any future modal use site consumes; no per-use-site focus-trap copy-paste.

### Negative

- Adds one runtime dependency: `@radix-ui/react-dialog`. Bundle impact ~12 kB gzipped on the routes that import it.
- Adds peer-dep alignment requirement: Radix dialog needs React 19 (already in use), no further upgrade pressure.

### Risks

- **Hydration mismatch on portal content**: SSR rendering of `Portal` differs from client. *Mitigation*: gate `<Dialog>` with `useEffect`-based mount flag in the wrapper, or use Radix's recommended `modal={true}` + suppress-hydration pattern. Standard practice; will be tested in the e2e suite under the missing-twin flow.
- **Future divergence from shadcn**: if the project later adopts shadcn proper, the hand-written wrapper may need refactoring to match. *Mitigation*: keep the wrapper close to shadcn's published Radix Dialog wrapper API so future adoption is a copy-paste, not a rewrite.

## Implementation Notes

- Install: `bun add @radix-ui/react-dialog`.
- Wrapper file: `app/components/ui/dialog.tsx`. Exports: `Dialog`, `DialogTrigger`, `DialogContent`, `DialogHeader`, `DialogTitle`, `DialogDescription`, `DialogFooter`, `DialogClose` (mirror shadcn's named export shape).
- Tailwind styling — follow the project's existing component aesthetic; no new design tokens.
- E2E test (Playwright) covers: modal open on click, focus trap, Escape closes (cancel path), Enter on confirm button triggers the redirect, return-focus to the originating menu item on close.
