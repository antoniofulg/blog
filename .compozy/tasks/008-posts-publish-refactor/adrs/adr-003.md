# ADR-003: Language-switcher UX = per-menu-item availability hint + confirm modal

## Status

Accepted (supersedes [ADR-002](adr-002.md))

## Date

2026-05-21

## Context

ADR-002 locked the missing-twin switcher UX as "modal-only for V1" — switcher button stays plain, modal is the first signal of a missing twin, pre-click hint deferred to V2. The council debate had treated the switcher as a single toggle button and weighted the "loader contract surface" risk heavily on that assumption.

During PRD clarification (Q-PRD2 round), the user reversed the V1 scope: surface the hint *in the language dropdown menu items* before the click, so the modal becomes a confirmation rather than a discovery. Codebase research surfaced a fact the council debate had missed: the switcher is not a toggle button. It is a `<DropdownMenu>` rendering one item per locale via `app/components/ui/language-menu.tsx`. Each item is a natural surface for a per-item availability state — disabled visual, accessible aria-disabled hint, label suffix like "(not available)" — without requiring any new layout-level state plumbing.

Additionally, Q-PRD3 scoped the hint logic to content routes only: posts (`PostEntry.hasTwin` already available) and static pages (`staticPageHasTwin` from ADR-001's encapsulated module). Structural routes (homepage, tag pages, 404) render both menu items as available with no hint, because they exist natively in both locales via the locale param. This bounds the loader contract risk that drove ADR-002's caution.

## Decision

The V1 language switcher uses **per-menu-item availability hint + confirm modal on click**.

- Each `<DropdownMenuItem>` in `language-menu.tsx` carries an `available: boolean`. When `false`, the item renders with a visual hint ("(not available)" suffix or analogous treatment) and `aria-disabled` semantics, but stays clickable.
- Clicking an available item navigates directly to the twin via TanStack Router `<Link params={(prev) => ({ ...prev, locale: targetLocale })}>`.
- Clicking an unavailable item opens a confirm modal in the **current page's language** (Q-PRD2): "This content is not available in [target language]. You will be redirected to the home page in [target language]. Continue?". Confirm → target-locale home; Cancel → stay.
- Availability is computed per route type:
  - Post routes: `PostEntry.hasTwin` from `app/lib/site-model.server.ts`.
  - Static page routes: `staticPageHasTwin(slug, targetLocale)` from `app/lib/mdx/pages.server.ts` (ADR-001).
  - All other routes (homepage, tag pages, 404): always `available = true`.
  - Admin: switcher hidden (locale-agnostic surface; see Q-PRD3 follow-up note in the PRD).

## Alternatives Considered

### Alternative 1: Modal-only (the superseded ADR-002 decision)

- **Description**: Switcher items always plain; modal is the first signal of a missing twin.
- **Pros**: Single implementation site; no contract for loaders to expose twin state.
- **Cons**: Surprise UX — readers click expecting a translation and get a modal; on a small bilingual corpus the modal fires on most clicks, training reflexive dismissal; product-mind in the council debate called this the highest-risk eroder of the bilingual investment.
- **Why rejected**: The contract risk that drove ADR-002 collapses once the switcher is recognized as a per-item menu (not a button); the surface is naturally per-item; the hint is now cheap; modal-only's UX cost no longer pays for itself.

### Alternative 2: Build-time twin manifest

- **Description**: Emit a static JSON manifest of all translated slugs at build time; switcher queries the manifest directly from the URL.
- **Pros**: One implementation site; works for all route types uniformly; no loader cooperation needed.
- **Cons**: Build-pipeline scope (manifest regeneration on every content change, cache invalidation); pays infrastructure cost ahead of validated need.
- **Why rejected for V1**: Pure scope inflation given that `PostEntry.hasTwin` and `staticPageHasTwin` already provide the data through the router context. Listed as the V2 escape hatch if loader-context plumbing ever becomes painful for new route types.

## Consequences

### Positive

- Reader sees twin availability before committing to a click — sets expectation, prevents modal-as-surprise.
- Modal remains the safety net for accidental clicks of unavailable items; supports the screen-reader and keyboard-navigation flow uniformly.
- Per-item state lives entirely inside `language-menu.tsx`; no new contract on every loader — the existing data sources (`PostEntry.hasTwin`, `staticPageHasTwin`) already cover content routes.
- Reader trust in the bilingual feature is reinforced even during corpus build-out when most posts will lack twins.

### Negative

- Two states per menu item to test (`available`, `not available`) instead of one — modest test surface increase.
- Hint copy needs i18n in both locales using the inline `Record<Locale, string>` pattern already in use in `language-menu.tsx:6-24`.

### Risks

- **Stale availability**: if the router context's twin data is stale (e.g., after a hot-reload during dev), the hint may misrepresent state. *Mitigation*: data flows from server loader on each navigation; SSR + hydration handles the staleness boundary already. Add a manual test step in the PRD acceptance: verify hint updates correctly after navigating between posts with different twin states.
- **Admin surface ambiguity**: admin is locale-agnostic but the dropdown is in the global header. *Mitigation*: hide the language-menu trigger when route matches `/admin/*` (decision recorded above; PRD captures this as F5 scope).
- **Hint copy review**: "(not available)" is one option; "(no translation)" is another; "(only in EN)" is a third. *Mitigation*: PRD captures copy as Open Question Q-O3 for resolution before merge.

## Implementation Notes

- `app/components/ui/language-menu.tsx` gains an `available` prop per item; the dropdown trigger uses the existing locale label.
- A single helper `getTwinAvailabilityForCurrentRoute()` resolves the per-route source (post / page / structural / admin).
- The confirm modal is the new primitive added under `app/components/ui/dialog.tsx` (or analogous) — see PRD F-MODAL feature.
- Test surface: `app/tests/header.test.ts` extended to cover both hint states; `tests/e2e/public-read.spec.ts` extended to cover en → pt-br direction (currently broken per `:77` documentation) plus the missing-twin modal confirm + cancel paths.
