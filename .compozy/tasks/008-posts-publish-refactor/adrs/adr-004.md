# ADR-004: Rollout = single release for V1

## Status

Accepted

## Date

2026-05-21

## Context

TASK-0008 V1 bundles six concerns: drop `published` flag, static-pages directory convention (ADR-001), language-switcher rewrite with menu-item hint + confirm modal (ADR-003), `sitemap.xml` with hreflang pairs (Q-PRD1), admin list + locale filter (Q-PRD6), and a new modal primitive in `app/components/ui/` (gap surfaced by PRD codebase research).

Three rollout shapes were debated during PRD approach selection:

- **A. Single release** — everything ships in one PR / one deploy.
- **B. Two releases by surface** — structural first (publish-flag + static-pages + admin), reader-facing second (switcher + modal + sitemap + hreflang).
- **C. Three stacked releases** — modal primitive + structural → switcher → sitemap + hreflang.

The blog has 2 e2e fixture posts and a single author. CI runs full Vitest + Playwright + lint + build on every PR. CD deploys on merge to main with no feature-flag system in place.

## Decision

V1 ships as a **single release** — one PR, one deploy, one task branch (`TASK-0008/posts-publish-refactor`).

## Alternatives Considered

### Alternative 1: B — Two releases by surface

- **Description**: Release 1 = structural (publish-flag + static-pages + admin trim). Release 2 = reader-facing (switcher + modal + sitemap + hreflang).
- **Pros**: Each PR ≤ ~15 files; structural changes bake before user-visible changes ship; static-pages module gets a release cycle to settle before the switcher consumes `staticPageHasTwin`.
- **Cons**: Two coordination cycles (PR → CI → merge → deploy → verify) for a single coherent refactor; the structural release is invisible to readers, which means the visible reader value is delayed; CI gates already prevent staged-release benefits from materializing on a single-author solo project with no rollback distinction between intermediate states.
- **Why rejected**: Coordination cost outweighs blast-radius benefit at this scale.

### Alternative 2: C — Three stacked releases

- **Description**: (1) Modal primitive + structural cleanup → (2) Switcher rewrite using the modal → (3) Sitemap + hreflang.
- **Pros**: Smallest per-PR blast radius; each PR has a tight scope and reviewable narrative.
- **Cons**: Three rounds of PR → CI → merge → deploy → manual smoke; the sitemap + hreflang work is functionally coupled to the switcher's twin-lookup helper (both consume `staticPageHasTwin` / `PostEntry.hasTwin`), so splitting them creates artificial seams the test suite has to mock; for a solo author with 2 fixture posts, this is over-decomposition.
- **Why rejected**: Over-decomposition for the scale of corpus and team.

## Consequences

### Positive

- One atomic narrative — the entire refactor lands or none of it does. Easier to communicate the change (single PR description, single CHANGELOG entry).
- The switcher's twin-lookup helper, the static-pages module, the sitemap's hreflang emitter, and the admin's locale filter all share data sources; landing them together means each consumer has its dependency available at the moment of use, no temporary shims.
- Single merge → single deploy → single set of verification steps. Lowest end-to-end overhead.

### Negative

- PR will be large — estimated ~25-35 files changed across DB schema, queries, routes, components, tests, sitemap generator, CI assertions.
- Review surface is wide; reviewer (or future-self) needs to context-switch between layers in one sitting.
- If a single concern needs to be reverted post-deploy, the rollback granularity is the whole PR, not the offending slice.

### Risks

- **Reviewer fatigue / review-quality dilution** for a large PR. *Mitigation*: structure the PR description with explicit per-concern sections + per-file commentary mapping each file to its concern; commit history within the branch keeps logical separation (one concern per commit ideally).
- **Pre-deploy verification overlooks edge case**. *Mitigation*: PRD's Success Metrics double as the manual smoke checklist before merge; e2e coverage extends to the new modal + hint + missing-twin paths before merge.
- **Migration risk** for the `isPublished` column drop. *Mitigation*: migration is generated by drizzle-kit and runs in CD's migrate-before-restart ordering (already enforced per `.agents/rules/cicd.md`); corpus is 2 fixture posts, no real data trapped behind the flag.
- **Rollback granularity**. *Mitigation*: GHCR image tagging (`:latest` + `:<short-sha>`) per `.agents/rules/cicd.md#ghcr-image-strategy` allows rollback to the prior `:<short-sha>` without rebuild; granularity is "before vs after this refactor", which is acceptable.

## Implementation Notes

- Branch: `TASK-0008/posts-publish-refactor` (already in use).
- Commits within the branch should be logically separable: one per concern (drop-flag, static-pages, switcher, sitemap, admin, modal-primitive). The merge is one PR, but the commits keep the per-concern story for reviewers and for post-deploy git-bisect.
- Pre-merge checklist (from PRD Success Metrics): full Vitest + Playwright pass; manual smoke per persona (P1 author, P2 bilingual reader); content-audit run shows no regression on translation-gap rule with `enumerateStaticPages` wired in.
