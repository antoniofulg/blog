# ADR-007: Sitemap.xml generated per-request, no cache

## Status

Accepted

## Date

2026-05-21

## Context

PRD F5 requires emitting `sitemap.xml` with reciprocated `<xhtml:link rel="alternate" hreflang>` pairs only when both locale variants exist. The codebase has no sitemap today. Three generation strategies were considered:

1. **Per-request, no cache** — `/sitemap.xml` route reads the post list from Drizzle + page list from `enumerateStaticPages` on every request, renders XML inline, returns.
2. **Per-request with in-memory cache** — cache the rendered XML in module-scope; invalidate on `indexer.ts` file-watcher events.
3. **Build-time static asset** — emit `public/sitemap.xml` during the build step; CD pipeline picks it up; CDN-cacheable.

User selected option 1 (per-request, no cache).

The blog has ~2 fixture posts today, ~50-100 posts at any plausible long-term scale, plus a handful of static pages. Crawl frequency from Googlebot for a personal blog is on the order of dozens of requests per day, not thousands per minute. Per-request generation against Drizzle + filesystem at this scale is microseconds-to-low-milliseconds.

## Decision

The `/sitemap.xml` route is implemented as `app/routes/sitemap[.]xml.ts` following the existing `app/routes/robots[.]txt.ts` content-type idiom. On every GET, the handler:

1. Calls a new `getSitemapEntriesFn` server fn that reads all posts (no `isPublished` filter — F1 removes it) and all static pages (via `enumerateStaticPages` for both locales).
2. For each entry, computes the locale twin via the same source the language menu uses (`PostEntry.hasTwin` for posts, `staticPageHasTwin` for pages).
3. Renders an XML string with `<urlset>` containing one `<url>` per entry; each `<url>` includes `<xhtml:link rel="alternate" hreflang="...">` only when the twin exists, plus `hreflang="x-default"` on the EN homepage entry.
4. Returns `new Response(xml, { status: 200, headers: { "content-type": "application/xml" } })`.

No caching layer. No CDN-specific cache headers beyond defaults (the response stays freely cacheable upstream if traffic patterns ever justify it — that becomes a separate decision, not this ADR's concern).

## Alternatives Considered

### Alternative 1: Per-request with in-memory cache invalidated by indexer

- **Description**: Module-scope `Map<Locale, string>` holds rendered XML. `indexer.ts` file-watcher events invalidate the cache; next request re-renders.
- **Pros**: Faster repeat hits (~0.1 ms instead of ~5 ms).
- **Cons**: New cross-module dependency (indexer → sitemap cache); cache-invalidation surface to test and debug; meaningful only at request volumes the blog will not see.
- **Why rejected**: Premature optimization; the cache-invalidation bug surface is more dangerous than the latency saved.

### Alternative 2: Build-time static asset

- **Description**: A build script enumerates posts + pages, writes `public/sitemap.xml`, CD picks it up via the existing build pipeline.
- **Pros**: Fastest possible response (static asset served by the runtime); CDN-cacheable indefinitely.
- **Cons**: Stale until next deploy (a new post added between deploys is missing from the sitemap for hours); requires CI step coupling; needs to handle posts added by hot-reload during dev separately.
- **Why rejected**: Staleness during the deploy window is a real concern for a content-frequent blog; per-request generation is fresh by construction.

## Consequences

### Positive

- The sitemap reflects the current content state on every request — no staleness window, no cache-invalidation bug surface.
- Matches the project's existing simplest-thing-that-works pattern (mirror of `robots[.]txt.ts`).
- The hreflang reciprocity logic lives in one place (`getSitemapEntriesFn`), making the F5 acceptance test (Success Metric #5) straightforward to assert.

### Negative

- Each request does a Drizzle query + a filesystem walk. At blog scale this is invisible; at orders-of-magnitude higher traffic it becomes a candidate for option 1's cache or option 2's static asset.

### Risks

- **Latency under unexpected crawl bursts**: a misbehaving crawler hammering `/sitemap.xml` hundreds of times per second could create artificial DB load. *Mitigation*: nginx / Traefik in front of the app already coalesces; standard rate-limiting middleware applies if the case ever arises; sitemap is a single read, not a write.
- **Hreflang asymmetry bug**: the per-request logic must correctly emit reciprocal hreflang pairs. *Mitigation*: dedicated unit test for `getSitemapEntriesFn` validates that for every emitted `hreflang` annotation, the referenced URL appears in the urlset with a reciprocal annotation (Success Metric #5 enforces this). Plus a `sitemap` finding category may be added to the app-audit skill in V2.

## Implementation Notes

- Route file: `app/routes/sitemap[.]xml.ts` (escaped `.` so TanStack Router treats the file basename literally — same convention as `app/routes/robots[.]txt.ts`).
- Server fn: `getSitemapEntriesFn` in a new `app/routes/sitemap[.]xml.server.ts` or co-located inline in the route file (under the ~80 LOC threshold). Returns `Array<{ loc: string; alternates: Array<{ hreflang: string; href: string }> }>`.
- XML rendering: hand-written string template (no XML lib dep). Escape attribute values with a small helper; the URL set is well-bounded so this is safe.
- Canonical origin: read from an env var (e.g., `SITE_URL`) with a sensible localhost default for dev. TechSpec captures this.
- Tests: Vitest unit test asserts reciprocity invariant + correct hreflang emission across mixed twin / no-twin fixtures.
