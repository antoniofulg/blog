# ADR-002: Language-switcher missing-twin UX = modal-only for V1

## Status

Superseded by [ADR-003](adr-003.md) — user reversed the V1 scope during PRD clarification to add a pre-click per-menu-item hint, keeping the modal as confirmation only.

## Date

2026-05-20

## Context

Q3 (locked): when a reader clicks the language switcher and the target-locale twin does not exist, show a confirm modal ("content not available in [target lang], you will be redirected to home page, proceed?"). On confirm → target-locale home. On cancel → stay.

Open question debated by the council: should the switcher button additionally surface a "(not translated)" hint *before* the click — making the modal a confirmation rather than a discovery — or keep the modal as the first signal?

- **B1 (pre-click hint)**: every page loader exposes `targetLocaleHasTwin` to the layout. Switcher button shows a marker before click; modal still confirms.
- **B2 (modal-only)**: switcher button is plain. Modal is the first signal.

`PostEntry.hasTwin` already exists for posts (`app/lib/site-model.server.ts:268`). For static pages, ADR-001 makes `staticPageHasTwin` available through the `pages.server.ts` module. For other routes (homepage, tag pages, 404, future routes), no twin concept applies cleanly — pre-click hint would require per-route boilerplate or a new cross-cutting layout loader.

Council opened 3-1 for B2. `product-mind` argued B1 on reader-trust grounds: "switcher will spend most of its life pointing at missing translations during corpus build-out; a modal-on-every-click trains readers to ignore the feature." `devils-advocate` partially conceded B1 *for `$slug` post routes specifically* (cheap, twin already known), holding firm against global B1. `product-mind` partially conceded B2 *for V1* after acknowledging the user population that experiences "surprise" as a problem does not yet exist (current corpus: 2 fixture posts), and proposed a build-time twin manifest as the V2 path if engagement data warrants it.

## Decision

V1 ships **B2 globally**: the switcher button shows no twin-state marker. On click, the switcher checks twin availability for the current route via the appropriate source (`PostEntry.hasTwin` for posts, `staticPageHasTwin` for pages, treat as "no twin" for non-content routes like homepage). If absent → confirm modal → home redirect or cancel. If present → direct navigation to the twin.

B1 is **explicitly deferred to V2** with documented trigger conditions (see Risks below).

## Alternatives Considered

### Alternative 1: B1 globally — pre-click hint on all routes

- **Description**: Every route loader exposes `targetLocaleHasTwin` to the layout; switcher button shows "(not translated)" or analogous marker.
- **Pros**: Sets reader expectation; reduces modal noise during corpus build-out; better perceived UX.
- **Cons**: Cross-cutting loader contract — every current and future route must compute the boolean; missing the contract silently lies; non-post routes (homepage, tag, 404) have no natural twin semantics, requiring per-route bespoke logic.
- **Why rejected**: The invisible-failure mode (loader forgets to expose the boolean → switcher shows incorrect state) trains worse user behavior than the modal-as-discovery flow. The user population whose pain B1 solves does not yet exist; building the contract pre-emptively pays the cost without validating the need.

### Alternative 2: B1 scoped to post routes only

- **Description**: Pre-click hint shown on `$slug` post routes only (where `PostEntry.hasTwin` is free); plain button + modal everywhere else.
- **Pros**: Captures the cheap case; no new loader contract; partial UX improvement.
- **Cons**: Inconsistent switcher behavior across routes (hint on one page, none on the next) is potentially more confusing than uniform modal-only behavior; adds a code path that V2 may obsolete via a unified manifest approach.
- **Why rejected**: Inconsistency cost outweighs the partial UX win for V1. Worth revisiting as part of V2 when a unified strategy can be chosen.

### Alternative 3: Build-time twin manifest (V2 candidate)

- **Description**: Emit a static JSON manifest of all translated slugs at build time; layout queries it directly from the URL — no loader cooperation needed.
- **Pros**: Neutralizes the invisible-failure mode; one implementation site; works for all route types uniformly.
- **Cons**: Build-pipeline scope; manifest must regenerate on every content change; cache invalidation concern.
- **Why rejected for V1**: Scope inflation. Listed as the recommended V2 path if engagement data justifies B1.

## Consequences

### Positive

- One implementation site (the switcher component); no per-route loader contract.
- Behavior consistent across every route, current and future.
- Removes the home-redirect bug (Q3) and the hardcoded `if`-chain in `app/components/layout/header.tsx:49-71` simultaneously.
- The modal makes the missing-translation case visible and dismissable, treating it as a discoverable state rather than silent navigation.

### Negative

- Reader who clicks the switcher hoping for a translation gets a modal instead of a direct navigation hint. For a corpus where most posts will start untranslated, the modal will fire frequently.
- No pre-click affordance — readers cannot scan the page and tell whether a translation exists without clicking.

### Risks

- **Modal fatigue**: if the modal fires on most switcher clicks and readers learn to dismiss it reflexively, the bilingual feature loses signal. *Mitigation:* V2 trigger conditions documented below.
- **V2 trigger conditions** (revisit B1 / manifest path when any apply):
  1. Total post count exceeds **10** AND fewer than 50% have twins.
  2. Switcher-click telemetry shows modal-dismiss rate > 60% over any 30-day window.
  3. A direct user report calls out the missing pre-click affordance.
- **Switcher implementation drift**: the switcher must query the *correct* twin source per route type (post vs page vs other). *Mitigation:* a single `getTwinAvailabilityForCurrentRoute()` helper that owns the routing logic; tests cover all three route categories.

## Implementation Notes

- Rip the hardcoded `if`-chain in `useLangSwitcher` (`app/components/layout/header.tsx:49-71`).
- Replace with: `<Link params={(prev) => ({ ...prev, locale: targetLocale })}>` (TanStack Router idiomatic form), wrapped in an `onClick` that intercepts when twin is absent and opens the modal.
- Twin lookup helper resolves: post route → `PostEntry.hasTwin`; static page route → `staticPageHasTwin(slug, targetLocale)` (per ADR-001); other routes → treat as "no twin" (modal always fires on click; on confirm, redirect to target-locale home).
- Emit `<link rel="alternate" hreflang>` only when twin exists (Search Engine Journal asymmetric-hreflang anti-pattern).
