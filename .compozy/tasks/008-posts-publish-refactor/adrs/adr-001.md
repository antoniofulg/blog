# ADR-001: Static-pages storage = filesystem-only, encapsulated module

## Status

Accepted

## Date

2026-05-20

## Context

TASK-0008 introduces a static-pages convention at `app/content/pages/<locale>/<slug>.mdx` (Q2, locked). Posts already live in Postgres via `app/db/indexer.ts` (file-watcher upserts rows); `PostEntry` exposes `hasTwin: boolean` through `app/lib/site-model.server.ts`. The open question for static pages was whether to extend the same DB indexer pipeline to them (A1) or keep them filesystem-only (A2).

The static-page corpus is expected to remain single-digit forever (about, uses, maybe 3 more — pages change with deploys, not content workflows). Two consuming surfaces already exist that need to enumerate or check page state: the future sitemap, and the existing `content-audit` skill (translation-gap parity rule applies to pages once the new layout lands).

Council debate (`pragmatic-engineer`, `architect-advisor`, `product-mind`, `devils-advocate`) opened 3-1 for A2 (filesystem-only) with `architect-advisor` defending A1 on boundary-debt grounds: "two resolution strategies for one semantic question (does a translation exist?) becomes accidental debt as content-adjacent features grow." Pragmatic-engineer partially conceded that risk and proposed the resolution: keep A2, but encapsulate page access behind a single module rather than sprinkling raw `existsSync` calls.

## Decision

Static pages stay **filesystem-only**. All page access — load, twin-check, enumeration — goes through a single `app/lib/mdx/pages.server.ts` module. No DB row, no indexer extension, no `content_type` discrimination.

The module exposes a stable surface:

- `loadStaticPage(slug, locale)` — read + parse + render (replaces `app/lib/mdx/about.server.ts`)
- `staticPageHasTwin(slug, targetLocale)` — boolean via `existsSync`
- `enumerateStaticPages(locale)` — directory walk for sitemap + audit consumers

Sitemap, content-audit, and any future content-discovery surface query this module, never raw `existsSync` or direct directory reads.

## Alternatives Considered

### Alternative 1: A1 — index static pages into DB

- **Description**: Extend `app/db/indexer.ts` to upsert pages as a separate `content_type`; expose pages through `site-model.server.ts` alongside posts.
- **Pros**: Single resolution strategy for "does this content exist?"; `hasTwin` tracking free; consistent query surface for sitemap + audit.
- **Cons**: DB write path for low-cardinality data that changes annually; indexer needs page-vs-post discrimination; `about.mdx` data migration; structural-URL availability risk — if indexer misses a deploy window, `/about` returns 404 silently (posts tolerate this, structural pages should not).
- **Why rejected**: Pays multi-author-platform costs to serve a corpus that will never need them. Structural availability risk specifically disqualifies it for pages whose URLs are part of the site contract.

### Alternative 2: A2 raw — filesystem checks scattered at call sites

- **Description**: Each consumer (sitemap, content-audit, switcher, route loader) calls `existsSync` directly.
- **Pros**: Trivially simple per call site.
- **Cons**: Architect's warning materializes — every new content-adjacent feature reinvents twin/enumeration logic; behaviors drift; one consumer eventually gets the locale path wrong.
- **Why rejected**: Encapsulation cost is one file; refactoring three drifted callers later is far more expensive.

## Consequences

### Positive

- Zero schema migration; static-pages PR ships without touching `app/db/schema.ts` or `indexer.ts`.
- `existsSync` semantics match the locked file-presence visibility gate from Q1.
- Structural-page URLs remain available regardless of indexer health.
- Single boundary (`pages.server.ts`) gives one place to upgrade if A1 ever becomes warranted.

### Negative

- Sitemap and content-audit grow a second enumeration path (posts via DB, pages via module). The encapsulation contains it, but the seam is documented and real.
- Static pages do not participate in `site-model.server.ts` `PostEntry` queries; any cross-content query must call both.

### Risks

- **Encapsulation leak**: a future contributor adds `existsSync('app/content/pages/...')` outside the module. *Mitigation:* add a lint rule or grep-based audit check that flags direct `app/content/pages` filesystem access outside `pages.server.ts`.
- **Promotion trigger missed**: if static pages grow past ~5 OR a new consumer needs queryable per-page metadata (status, analytics, scheduled publishing), A1 becomes the right answer. *Mitigation:* document the trigger explicitly in `pages.server.ts` header comment and revisit in a future ADR.

## Implementation Notes

- Move `app/content/<locale>/about.mdx` → `app/content/pages/<locale>/about.mdx`.
- Replace `app/lib/mdx/about.server.ts` with the generalized `app/lib/mdx/pages.server.ts`.
- Update `app/routes/{-$locale}/about.tsx` to call `loadStaticPage("about", locale)` from the new module.
- Wire `enumerateStaticPages` into the content-audit pipeline so translation-gap parity applies to pages identically to posts.
- The switcher consumes `staticPageHasTwin` from this module (not from `PostEntry.hasTwin`, which is post-only).
