# ADR-003: MDX Compilation Approach — @mdx-js/mdx Direct

## Status

Accepted

## Date

2026-05-02

## Context

The scaffold must compile `.mdx` files server-side per request without shipping the MDX compiler to the client bundle (PRD constraint). Three approaches were evaluated: using `@mdx-js/mdx` directly with `compile()` + `run()`, adapting `next-mdx-remote` from the Next.js ecosystem, and using a raw `unified` + remark/rehype pipeline that produces an HTML string instead of a React component tree.

## Decision

Use `@mdx-js/mdx` directly via its `compile()` and `run()` functions inside a TanStack Start server function (`app/lib/mdx.server.ts`). Compile with `outputFormat: 'function-body'` and run against `react/jsx-runtime`. Shiki is wired in as a rehype plugin (`@shikijs/rehype`) at compile time. The resulting `Content` component is returned from the server function and rendered server-side — the MDX runtime and compiler never appear in any client bundle.

## Alternatives Considered

### Alternative 1: next-mdx-remote (adapted)

- **Description**: Use `serialize()` + `<MDXRemote>` from `next-mdx-remote`, adapting it away from its Next.js server actions pattern.
- **Pros**: Familiar API with built-in frontmatter parsing; popular community adoption
- **Cons**: Introduces a Next.js-flavored dependency into a non-Next.js scaffold; `<MDXRemote>` is designed for client hydration which conflicts with the server-only constraint; adaptation adds complexity and fragility
- **Why rejected**: The dependency name signals Next.js to future readers; the client-hydration design must be actively fought rather than used as intended.

### Alternative 2: unified/remark/rehype → HTML string (no MDX runtime)

- **Description**: Use `unified()` + `remark-parse` + `remark-rehype` + `rehype-stringify` to convert MDX to a raw HTML string. Render with `dangerouslySetInnerHTML`.
- **Pros**: Simplest bundle boundary — zero React runtime involvement; lightest dependency set
- **Cons**: Loses the ability to use React components inside `.mdx` files (e.g., custom callout components, interactive demos); `dangerouslySetInnerHTML` is less type-safe; MDX's JSX support is its primary differentiator over plain Markdown
- **Why rejected**: Eliminating React component support inside MDX files removes the main reason to use `.mdx` over `.md`. A personal blog scaffold should support custom MDX components.

## Consequences

### Positive

- Full React component support inside `.mdx` files (import and use any React component)
- No client bundle pollution — `@mdx-js/mdx` and `react/jsx-runtime` imports are in `mdx.server.ts` which is never included in client chunks
- Shiki integrates cleanly as a `rehypePlugin` at compile time — syntax highlighting is baked into the HTML before it leaves the server
- Direct API gives full control over the remark/rehype plugin chain

### Negative

- `compile()` + `run()` is a two-step API that is less familiar than a single `serialize()` call
- Compiled output is a function-body string that must be `run()` against the correct runtime — wrong runtime reference causes silent hydration failures

### Risks

- **`vite-env-only` import guard**: `mdx.server.ts` must be listed in `vite-env-only` config to prevent Vite from accidentally bundling it for the client during development hot-reload
- **`react/jsx-runtime` version mismatch**: The `run()` call must use the same React version as the application — ensured by importing from the same `react` package in both places

## Implementation Notes

```typescript
// app/lib/mdx.server.ts
import { compile, run } from '@mdx-js/mdx'
import * as runtime from 'react/jsx-runtime'
import remarkGfm from 'remark-gfm'
import rehypeShiki from '@shikijs/rehype'

export async function renderMdx(source: string) {
  const compiled = await compile(source, {
    outputFormat: 'function-body',
    remarkPlugins: [remarkGfm],
    rehypePlugins: [[rehypeShiki, { theme: 'github-dark' }]],
  })
  const { default: Content } = await run(compiled, { ...runtime, baseUrl: import.meta.url })
  return Content
}
```
