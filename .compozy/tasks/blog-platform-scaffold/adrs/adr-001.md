# ADR-001: Scaffold Scope — Full Starter Kit

## Status

Accepted

## Date

2026-05-02

## Context

The scaffold targets a personal content/blog platform use case. During the council debate, three scope levels were considered: (1) static MDX with no database or auth, (2) public blog pages with MDX only, and (3) a full starter kit including public pages, Drizzle-backed posts, Better Auth admin protection, and CRUD admin interface. A parallel concern surfaced: the scaffold stacks four relatively new tools (TanStack Start v1, React 19, Better Auth, Drizzle) simultaneously, creating a debugging surface with limited community precedent.

## Decision

Ship V1 as a full starter kit — public blog list and detail pages, MDX rendering (server-side only), Drizzle ORM schema for posts with migrations, Better Auth wired for admin route protection, and a basic admin CRUD interface. The maturity risk is mitigated by baking the following guard rails directly into the scaffold: pinned lockfile, explicit version constraints for all four immature dependencies, an auth-to-database smoke test that verifies the integration round trip, and a known-good Docker Compose baseline tested against a clean clone.

## Alternatives Considered

### Alternative 1: Static MDX (no database, no auth)

- **Description**: Blog content lives as MDX files in the repo; no Postgres, no Better Auth, no admin UI
- **Pros**: Simplest possible path; zero integration risk; deploy to any static host
- **Cons**: Eliminates Postgres and Better Auth from the scaffold entirely, making it useless as a foundation for dynamic content projects
- **Why rejected**: Contradicts the user's stated goal of a full starter kit for database-backed blog projects

### Alternative 2: Public pages + MDX only (no auth, no admin)

- **Description**: Ships public-facing routes and MDX rendering with Drizzle-backed posts, defers Better Auth and admin CRUD to V1.1
- **Pros**: Smaller integration surface; easier to validate the 60-second DX promise
- **Cons**: Auth routes must be retrofitted later, creating an integration seam at a worse time; the scaffold is incomplete as a reference
- **Why rejected**: Deferral creates more integration debt than it avoids; the user explicitly chose the full kit scope

### Alternative 3: Bare tooling only

- **Description**: Just the five tools wired together — no demo pages, no auth, no example queries
- **Pros**: Maximum flexibility; zero opinionation beyond tooling
- **Cons**: Provides no working example of how the tools integrate; defeats the purpose of a scaffold
- **Why rejected**: A scaffold with no working integration examples offers no value over the official getting-started docs

## Consequences

### Positive

- Complete scaffold eliminates all setup friction for future content projects
- Auth and data layer integration done once, correctly, never retrofitted
- Forces explicit decisions about MDX compilation boundary and Drizzle client/server split before they can become bugs

### Negative

- Larger integration surface means more things that must work in harmony for the 60-second DX promise to hold
- Debugging interactions between TanStack Start + Better Auth + React 19 has limited community precedent

### Risks

- **MDX compilation leaking to client bundle**: Mitigated by using `vite-env-only` and restricting MDX to server-side compilation only
- **Better Auth plugin ordering**: Mitigated by placing `tanstackStartCookies` last in the plugin array as documented
- **Drizzle/postgres.js import leaking to client**: Mitigated by explicit server-only module externalization
- **Stacked immaturity causing opaque failures**: Mitigated by pinned lockfile, explicit version constraints, and an auth-to-database smoke test

## Implementation Notes

- Use `postgres.js` (not `pg`) as the Drizzle driver for Bun compatibility
- Route handler for Better Auth at `app/routes/api/auth/$.ts`
- MDX processing must happen server-side per request, never compiled on the client
- Admin seed script (initial admin user) must be part of the Docker Compose startup sequence to honor the 60-second promise
