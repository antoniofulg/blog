# ADR-004: File Watcher — Bun Native fs.watch

## Status

Accepted

## Date

2026-05-02

## Context

The content indexer (ADR-002) requires a file watcher to detect new, modified, and deleted `.mdx` files in the `/content` directory and trigger re-indexing without a manual sync command. Two options were evaluated: the `chokidar` package (used internally by Vite) and Bun's native `fs.watch` from the Node.js compatibility layer.

## Decision

Use Bun's native `fs.watch` (`import { watch } from 'node:fs'`) with `{ recursive: true }` to monitor the `content/` directory. Events are `'rename'` (file added or deleted) and `'change'` (file modified). Both trigger a full re-read of the affected file path and upsert into Postgres. The `bun run sync` script remains available as a manual fallback for CI and restricted environments (as specified in ADR-002).

## Alternatives Considered

### Alternative 1: chokidar

- **Description**: `chokidar.watch('content/**/*.mdx', { ignoreInitial: false })` emits granular `add`, `change`, and `unlink` events with built-in debouncing and cross-platform normalization.
- **Pros**: Mature cross-platform behavior (FSEvents on macOS, inotify on Linux, ReadDirectoryChangesW on Windows); explicit `add`/`change`/`unlink` events simplify handler logic; used by Vite, Webpack, and Jest — well-understood behavior
- **Cons**: Adds a runtime npm dependency (`chokidar` + `fsevents` optional peer); slightly heavier than a built-in alternative
- **Why rejected**: Bun native `fs.watch` with `recursive: true` covers the required events for a personal dev tool. Adding a dependency for a use case that the runtime handles is unnecessary complexity for a single-developer local scaffold. If cross-platform issues surface (Windows particularly), switching to chokidar is a low-risk upgrade.

## Consequences

### Positive

- Zero additional runtime dependency
- `fs.watch` is available in Bun without any install step
- Sufficient for the primary use case: macOS and Linux development environments

### Negative

- `fs.watch` events are less granular: `'rename'` covers both file creation and deletion; the handler must stat the path to distinguish them
- Recursive watching behavior on Windows is not guaranteed in Bun's current `fs.watch` implementation
- No built-in debouncing — rapid saves (e.g., editor autosave) may trigger multiple re-index calls for one file; mitigated with a simple setTimeout debounce in the handler

### Risks

- **Windows compatibility**: Not a priority for this personal scaffold (primary environment is macOS/Linux), but documented as a known limitation
- **Silent watcher failure**: If `fs.watch` fails to start (e.g., file descriptor limit), the admin goes stale with no user-visible error; mitigated by logging watcher start and emitting a warning if no `add` events fire within 5 seconds of startup
- **Debounce needed**: Handler wraps each event in a 100ms debounce to prevent duplicate upserts on rapid edits

## Implementation Notes

```typescript
// app/lib/watcher.server.ts
import { watch } from 'node:fs'
import { stat } from 'node:fs/promises'
import { indexFile, removeFromIndex } from '../db/indexer'

export function startContentWatcher(contentDir: string) {
  const debounceMap = new Map<string, ReturnType<typeof setTimeout>>()

  watch(contentDir, { recursive: true }, (event, filename) => {
    if (!filename?.endsWith('.mdx')) return

    const filePath = `${contentDir}/${filename}`
    clearTimeout(debounceMap.get(filePath))

    debounceMap.set(filePath, setTimeout(async () => {
      try {
        await stat(filePath)
        await indexFile(filePath)
      } catch {
        await removeFromIndex(filePath)
      }
    }, 100))
  })

  console.log(`[watcher] Watching ${contentDir} for .mdx changes`)
}
```
