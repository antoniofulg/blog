# ADR-002: Content Model and Sync Strategy

## Status

Accepted

## Date

2026-05-02

## Context

The PRD brainstorming session required a decision on two tightly coupled product questions: (1) where post content lives and how authors write it, and (2) how that content stays in sync with the Postgres index that powers public routing and admin queries. Three content models were considered. The answer drives the admin UI's role, the database schema, and the developer writing workflow.

## Decision

Use a **hybrid content model with an auto-sync indexer**: `.mdx` files in a `/content` directory are the canonical source of truth for post content. On `bun dev` start, a file watcher indexes all `.mdx` files into Postgres as drafts (recording slug, title, publishedAt, and view count). File changes are picked up automatically without a manual sync command. Draft/published status is controlled exclusively through the admin UI — not through frontmatter — so the file system and the database each own one dimension: content and publish state respectively.

## Alternatives Considered

### Alternative 1: Database-backed browser editor

- **Description**: Posts are stored entirely in Postgres. Authors write MDX in a browser textarea. No `/content` directory.
- **Pros**: Single source of truth; admin is a full CMS; works without a code editor
- **Cons**: Requires a rich MDX editor in the browser; loses the author's native editor tooling (syntax highlighting, snippets, Git history); adds significant admin UI complexity
- **Why rejected**: The developer's native code editor is a better MDX authoring surface than any browser textarea. This scaffold is personal-use; there is no need to write in the browser.

### Alternative 2: Explicit sync command (`bun run sync`)

- **Description**: Files are indexed only when the author runs `bun run sync`. Admin shows an out-of-sync badge when file system differs from DB.
- **Pros**: DB state is explicitly controlled; no unexpected background changes
- **Cons**: Requires an extra command after every new or edited file; breaks the zero-friction DX promise; out-of-sync state adds cognitive overhead
- **Why rejected**: An extra required command contradicts the 60-second DX goal. The auto-sync watcher provides the same predictability without the friction.

### Alternative 3: Frontmatter-driven, read-only admin

- **Description**: `draft: true` in frontmatter controls visibility. Admin is a read-only dashboard. No publish action in the UI.
- **Pros**: Simplest possible system; no write path in the admin
- **Cons**: Publishing requires editing a file, saving, and restarting the server; eliminates the value of a publish-control admin UI
- **Why rejected**: The user explicitly wants admin-controlled publish/unpublish. A read-only admin removes that capability.

## Consequences

### Positive

- Authors write MDX in their native editor with full tooling (Git history, syntax highlighting, snippets)
- New files appear in the admin automatically — no manual sync step
- Content is always recoverable from the file system; Postgres is a derived index, not the source of truth
- Admin's publish control is clean and independent of file edits

### Negative

- Two systems must stay semantically consistent: file content and DB publish state
- File renames or deletes must be handled gracefully by the indexer (slug re-mapping, orphan cleanup)
- The file watcher adds a startup dependency; if it fails silently, the admin goes stale

### Risks

- **File rename / slug collision**: Mitigated by using the file path (not frontmatter slug) as the stable identifier in Postgres
- **File watcher cross-platform reliability**: Mitigated by supporting a manual `bun run sync` as a fallback for CI or restricted environments
- **Draft state confusion**: Posts are always indexed as drafts on first discovery; frontmatter `draft` field is ignored in favor of DB state to keep the two systems from diverging

## Implementation Notes

- Index table columns: `id`, `file_path`, `slug`, `title`, `published_at`, `is_published`, `view_count`, `indexed_at`
- Publish state lives in DB (`is_published`); content always read from file at render time
- Watcher uses Node.js/Bun's native `fs.watch` or `chokidar` — decision deferred to TechSpec
