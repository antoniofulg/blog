# ADR-001: CI/CD V1 Scope — Pipeline-First, Standards-Included

## Status
Accepted

## Date
2026-05-05

## Context

The blog has no automated deployment or quality gate infrastructure. Every deploy is manual. The developer wants to introduce CI/CD with GitHub Actions, enforce Conventional Commits, and adopt a TASK-prefix branch naming convention tied to the compozy task system.

Council advisors debated whether commit/branch linting is justified for a single-author project. Two of three advisors recommended deferring conventions until the project has collaborators; the architect argued the TASK prefix creates compounding traceability value only when enforced consistently from day one.

An unasked-but-critical gap surfaced: no proposed scope included a migration step before the container restarts, which is the most common source of silent production failures in Docker-based deployments.

## Decision

Ship the complete scope in V1:

1. **GitHub Actions CI pipeline** — `make test && make lint && make check` on every push and PR
2. **GitHub Actions CD pipeline** — on push to `main`: build Docker image → push to GHCR → SSH to VPS → run `make db-migrate` → restart app container
3. **Conventional Commits enforcement** — `commitlint` in Lefthook `commit-msg` hook + CI check on PRs
4. **TASK-prefix branch naming** — `TASK-XXXX/short-description` enforced by GitHub Rulesets + PR workflow check
5. **Migration step before deploy** — explicit, ordered, fails loudly if it errors

Zero-downtime tooling (docker-rollout) is deferred — `docker compose up -d --no-deps` is sufficient for a personal blog. Compozy folder renaming is a separate task.

## Alternatives Considered

**Alternative A: Pipeline only, no conventions**
Drop commitlint and branch naming entirely. Ship the CI/CD pipeline in 1 task. Simple. The Pragmatic Engineer's recommendation.

Trade-off: User explicitly requested the conventions. Skipping them now means revisiting later, potentially with a messier commit history to contend with.

**Alternative B: Pipeline + conventions + folder rename in one task**
Bundle everything including renaming existing `.compozy/tasks/` folders to TASK-prefix format.

Trade-off: Renaming existing folders is a refactoring concern, not a CI/CD concern. Mixing them inflates scope and creates potential for half-finished state if the task stalls.

**Alternative C: Defer commit conventions until collaborators join**
Wait until there's a second author before enforcing commit standards.

Trade-off: Retroactively enforcing conventions on an existing commit history is harder than establishing them from the first commit. Cost of doing it now is low; cost of retrofitting grows.

## Consequences

**Positive:**
- Every push to main is automatically tested, built, and deployed — zero manual steps
- Commit history is structured from the start — tools like semantic-release, auto-changelog, and AI-assisted PR summaries can parse it
- Branch names trace back to compozy tasks — full thread from idea → task → branch → deploy
- Migration failures are surfaced before the container restarts, not after

**Negative:**
- Commitlint adds a local hook that can block commits — solo developer will occasionally fight it on quick fixes
- TASK-prefix branches require knowing the task number before starting work — minor ceremony overhead
- GitHub Secrets must be configured for VPS credentials, GHCR auth — one-time setup cost

## Open Questions

- Should the `commit-msg` hook have a bypass flag (e.g., `--no-verify`) documented for emergency hotfixes?
- What happens to branches created before the Ruleset is active? Retroactive enforcement is not possible; document the cutover date.
