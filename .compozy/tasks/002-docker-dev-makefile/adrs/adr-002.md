# ADR-002: Full V1 Scope — Complete Makefile + Dockerfile in Single Delivery

## Status

Accepted

## Date

2026-05-05

## Context

The docker-dev-makefile feature can be delivered at different scope levels:

- **Full V1**: All 10 target groups (setup, dev, dev-docker, build, preview, db-*, quality gates, lifecycle, deploy stub) shipped in one PR
- **Critical path first**: Only setup + dev + quality gates + db targets, with build/preview/lifecycle/deploy deferred
- **Makefile-only**: Thin wrappers over existing package.json scripts, no new Dockerfile or compose changes

The goal is to reduce contributor onboarding friction from clone to running state. A partial implementation risks delivering a Makefile that still requires manual steps, undermining the stated "≤ 2 commands" success criterion.

## Decision

Ship all 10 target groups in a single V1 delivery. The Makefile, multi-stage Dockerfile, and extended docker-compose.yml land together.

## Alternatives Considered

### Alternative 1: Critical path first

- **Description**: Setup + dev + quality gates + db targets only; build/preview/lifecycle/deploy in a follow-up PR
- **Pros**: Faster to ship; lower blast radius per PR
- **Cons**: Leaves the project in a partial state where `make build` and `make deploy` don't exist; contributors may work around the missing targets; follow-up PRs often don't ship
- **Why rejected**: A partial Makefile is a leaky abstraction — contributors hit missing targets and fall back to raw commands, defeating the purpose

### Alternative 2: Makefile-only, no Dockerfile

- **Description**: Thin Makefile wrapping existing package.json scripts; no Docker changes
- **Pros**: Zero Dockerfile risk; fastest to implement
- **Cons**: Contributors still need Bun + Postgres installed locally; does not reduce environment setup friction; doesn't solve the onboarding problem
- **Why rejected**: Doesn't address the core problem statement — open-source contributor dropout from multi-step environment setup

## Consequences

### Positive

- Makefile is complete and honest from day one — no missing targets
- Contributors get a full mental model of all available developer actions via `make help`
- CI integration can immediately validate `make test`, `make lint`, `make build` on every PR
- Production build path validated locally before any deploy

### Negative

- Larger PR surface area — Makefile + Dockerfile + docker-compose changes together
- More upfront effort than phased approach

### Risks

- Scope creep temptation during implementation — mitigated by YAGNI and the explicit out-of-scope list (devcontainer, CI/CD wiring, Windows native, secrets management beyond .env)

## Implementation Notes

- All targets are .PHONY
- make help uses ## comment scraping (awk pattern)
- Deploy stub for VPS: prints SSH/rsync instructions or calls a deploy.sh script; actual VPS tooling determined at implementation time
