# ADR-003: GHCR Image Tagging — Git SHA + latest

## Status
Accepted

## Date
2026-05-05

## Context

The CD pipeline builds a Docker image on every push to `main` and pushes it to GitHub Container Registry (GHCR). The team must decide how to tag the image so that the VPS can always pull the current version while also preserving traceability and rollback capability.

## Decision

Push each image with two tags:

1. **Git SHA tag** — the full 7-character short SHA of the triggering commit (e.g., `ghcr.io/owner/blog:abc1234`). Immutable and traceable: given any SHA tag, you can find the exact commit and workflow run that produced it.
2. **`latest` tag** — always points to the most recently built image from `main`. The VPS deploy script pulls `latest` without needing to know the current SHA.

## Alternatives Considered

**Alternative A: `latest` only**
Simplest approach — every build overwrites the single tag. Rejected because:
- No rollback path: to revert, you must rebuild and redeploy from a previous commit
- No traceability: inspecting the running container gives no information about which commit it was built from

**Alternative B: Date-stamped tags (`2026-05-05`) + `latest`**
Human-readable tags. Rejected because:
- Multiple builds on the same day would overwrite the date tag
- Not tied to a specific commit — cannot trace a running image back to source code without additional metadata

## Consequences

**Positive:**
- Every image is immutable and traceable to its source commit via the SHA tag
- Rollback is possible without rebuilding: update `latest` to point to an earlier SHA tag via `docker tag` + `docker push`
- The VPS always pulls `latest` — no need to thread the SHA through the SSH deploy step

**Negative:**
- GHCR accumulates one SHA-tagged image per deploy; old images need periodic cleanup (automated in V2 via a `docker image prune` cron or a cleanup workflow)
- Two push operations per build instead of one (negligible overhead)
