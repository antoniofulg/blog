# ADR-002: Pipeline Architecture — Two Separate Workflows (CI + CD)

## Status
Accepted

## Date
2026-05-05

## Context

The team must decide how to structure the GitHub Actions pipeline across workflow files. The options are a single unified workflow, two separate workflows, or two workflows with a manual approval gate before production deploy.

The blog is a single-author project. Every deploy fires against a live VPS with no staging environment. The developer expects to push to main and have the site updated within 5 minutes without any manual intervention.

## Decision

Use two separate workflow files:

- `ci.yml` — triggers on every push and pull request; runs `make test`, `make lint`, `make check` in parallel; blocks merge if any step fails
- `cd.yml` — triggers on push to `main` only; runs after `ci.yml` passes; builds the Docker image, pushes to GHCR, SSHes into the VPS, runs migrations, restarts the app container

Both workflows are fully automatic — no manual approval gates.

## Alternatives Considered

**Alternative B: Single workflow (`ci-cd.yml`)**
One file handles CI jobs on every push and a conditional deploy job (`if: github.ref == 'refs/heads/main'`). Same behavior as the chosen approach. Rejected because:
- GitHub's status UI shows one status badge for both CI and CD — harder to distinguish a failed deploy from a failed test at a glance
- The single file grows longer and harder to read as each pipeline evolves independently

**Alternative C: Two workflows with manual approval gate**
CD requires clicking "Approve" in GitHub Environments before the deploy step fires. Rejected because:
- Adds ceremony that defeats the goal of push-and-walk-away automation
- A solo developer approving their own deploy is redundant gate theater
- Any confidence from a manual gate already comes from the CI checks passing

## Consequences

**Positive:**
- Each workflow file has a single responsibility — easy to debug and iterate independently
- GitHub PR view shows distinct CI status checks (ci.yml) and deploy status (cd.yml) separately
- `cd.yml` can be disabled independently without touching the quality gate workflow
- Future extension (e.g., staging deploy, performance tests) adds a third workflow file without modifying the existing two

**Negative:**
- Two files to maintain vs. one
- Deploy can fire immediately after a push to main with no human gate — requires trust in CI coverage
