# ADR-002: 3-Phase Rollout for Site Restructure V1

## Status

Accepted

## Date

2026-05-13

## Context

V1 scope is fixed by ADR-001 — 8 features spanning deletions, route restructure, About MDX migration, typed i18n contract, robots.txt, fixture move, and test rewrites. The remaining open question is how to sequence delivery: a single-ship atomic cutover, a 2-phase delivery, or a finer-grained 3-phase delivery. The user-facing implication is how many discrete changes a visitor sees and how many review cycles the work passes through.

Context factors:
- WIP site, no production traffic — rollback theater is not required.
- User has explicitly stated preference for granular phasing (selected approach C during PRD brainstorming).
- The work spans three distinct concern areas: pure subtraction (deletions + broken-link cleanup), URL surface (routing + hreflang), and content shape (About + i18n contract).
- TanStack Router's `routeTree.gen.ts` regenerates after route changes; each phase that touches routes triggers a regeneration.

## Decision

Adopt a 3-phase rollout where each phase is independently shippable and internally consistent:

### Phase 1 — Slim down

Pure subtraction. No new features, no URL changes that affect existing content paths.

- Delete: `app/routes/tutorials.tsx`, `tutorials.$seriesSlug.tsx`, `projects.tsx`, `newsletter.tsx`, `search.tsx`, `app/components/tutorial-step.tsx`.
- Update header `NAV_LABELS` to remove `Tutorials` and `Projects`.
- Update footer `navLinks` and `resourceLinks` to remove links to deleted routes and broken paths (`/feed.xml`, `/sitemap.xml`).
- Ship `/robots.txt` as static route (allow all, canonical host).
- Regenerate `routeTree.gen.ts`.

Visitor experience: a tighter site with no stub pages, no broken footer links. Existing `/`, `/blog`, `/<slug>` redirects still work — URL behavior unchanged for content paths.

### Phase 2 — Route restructure

URL surface changes.

- Move blog post feed from `/$lang/blog` to be the locale root (`/` for English, `/pt-br/` for Portuguese).
- Delete `app/routes/blog.tsx`, `index.tsx`, `$slug.tsx` redirect shims.
- Adopt post URLs `/<slug>` (English) and `/pt-br/<slug>` (Portuguese).
- Render hreflang `<link rel="alternate">` pairs on every locale-aware page (feed root, post detail, About once Phase 3 lands).
- Default-locale `/` always renders English; no Accept-Language auto-redirect.

Visitor experience: cleaner URLs, post feed at the root, locale-prefix routing. About still at the legacy `/about` route until Phase 3.

### Phase 3 — Content and polish

Organic content + i18n contract + cleanup.

- Migrate About to MDX per locale: `content/en/about.mdx`, `content/pt-br/about.mdx`. Indie-dev minimal structure: identity 1-liner + photo + bio body + Now section + contact links. Free body prose, typed Zod frontmatter (required: `title`, `locale`; structured optional: `links[]`).
- Add `app/lib/i18n/strings.ts`: typed `Record<Locale, UIStrings>` + Zod schema. Populate only V1 keys (locale switcher labels in own-language form `English`/`Português`, post meta labels, locale-aware 404 copy).
- Move `content/en/lorem-ipsum.mdx` to `app/tests/fixtures/lorem-ipsum.mdx`. Update test references.
- Rewrite `public-routes.test.ts:154` and `header.test.ts:149` for locale-aware About routing.
- Apply locale-aware hreflang to About once it ships at `/about` + `/pt-br/about`.

Visitor experience: relabeled locale switcher (own-language), real About content per locale, V2 i18n extension path established (additive, not retrofit).

## Alternatives Considered

### Alternative 1: Single-ship V1 (atomic cutover)

- **Description**: All 8 features land in one PR/branch. Visitor sees the full restructure in one cutover.
- **Pros**: Atomic transition; no transitional UX states; one review cycle; one deploy.
- **Cons**: Large diff (~20-30 files touched); harder to isolate review attention; rollback only at the V1 level, not per concern area.
- **Why rejected**: User explicitly preferred granular phasing. Atomic cutover offers no real upside for a WIP site without production traffic; review cost is real and grows non-linearly with diff size.

### Alternative 2: 2-phase V1

- **Description**: Phase 1 = slim down. Phase 2 = restructure + content + polish (everything else).
- **Pros**: Two coherent visitor-facing improvements; balanced PR sizes.
- **Cons**: Phase 2 still bundles three distinct concerns (routes + content + i18n contract); review focus is split.
- **Why rejected**: User selected C (3-phase) over B (2-phase). The marginal review-attention cost of one extra phase is small; the clarity of separating routing from content shape is high.

## Consequences

### Positive

- Each phase has a clear visitor-facing outcome — Phase 1 = honest site, Phase 2 = clean URLs, Phase 3 = real content.
- Reviewers can focus on one concern area at a time (subtraction, then routes, then content).
- Per-phase rollback is granular; if Phase 2 reveals a routing collision, Phase 1's wins are unaffected.
- Phase 1 ships fastest (likely 1-2 days from PR open to merge) — immediate credibility uplift.
- Phase 3 carries the highest design surface (About, i18n contract); doing it last with the routing foundation already in place reduces context-switching during review.

### Negative

- Three PRs to coordinate instead of one; some overhead per PR (changelog entries, deploy cycles).
- `routeTree.gen.ts` regenerates twice (Phase 1 and Phase 2).
- Visitor sees three discrete changes over the V1 timeline rather than one.

### Risks

- **Branch divergence**: If multiple phases stack on top of each other before merge, rebase conflicts can compound. Mitigation: Phase 2 starts only after Phase 1 merges; Phase 3 starts only after Phase 2 merges.
- **Phase 2 routing collision**: TanStack Router optional-param idiom may interact awkwardly with `$lang.tsx`. If discovered mid-phase, the affected branch may need rework. Mitigation: prototype the routing primitive during Phase 1 in a scratch branch.
- **Phase 3 indexer changes**: Extending the indexer for About could affect existing post indexing. Mitigation: keep About indexing isolated (separate function or sibling module), validate posts still index correctly.

## Implementation Notes

- Branch naming per phase: `TASK-XXXX/phase-1-slim-down`, `TASK-XXXX/phase-2-route-restructure`, `TASK-XXXX/phase-3-content-i18n` (or equivalent compozy task numbering once assigned).
- Each phase ships its own changelog entry via the existing CI changelog automation.
- CD pipeline (`cd.yml`) runs after each phase merges to main; no special coordination needed beyond the existing workflow.
- All phases together remain within the V1 boundary as defined in ADR-001 — this ADR only governs sequencing, not scope.
