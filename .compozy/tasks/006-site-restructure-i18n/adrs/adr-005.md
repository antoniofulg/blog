# ADR-005: Cookie-First SSR Auto-Redirect on `/`

## Status

Accepted (supersedes the "Accept-Language auto-redirect" rejection inside ADR-001)

## Date

2026-05-14

## Context

ADR-001 listed "Accept-Language auto-redirect on `/`" as a rejected alternative on the grounds that auto-redirect adds cache-hostile complexity and breaks bot indexing. During TechSpec brainstorming, the author confirmed they want Portuguese-locale browsers automatically routed to `/pt-br/` on first visit, so visitors land in their language without manual switcher interaction.

The rejection in ADR-001 was correct for the naive form ("always Accept-Language") but did not consider the cookie-first variant. This ADR records the more careful design that the author actually wants.

Existing codebase support:
- `app/lib/locale.tsx` already exports `detectLocaleFromRequest(request)` that checks `locale` cookie first, then falls back to Accept-Language. Returns `Locale`.
- `LocaleProvider` already sets the `locale` cookie when `setLocale(...)` is called from the locale switcher.

## Decision

Implement a cookie-first SSR redirect at the root route (`/`):

1. On every SSR request to `/` (the default-locale post feed), the route's `beforeLoad` reads the request via `getRequest()` and calls `detectLocaleFromRequest(request)`.
2. If the `locale` cookie is set, respect it without redirect — the cookie is the persisted user choice.
   - Cookie value `"en"` → render the en feed at `/` (no redirect).
   - Cookie value `"pt-br"` → 302 redirect to `/pt-br/`.
3. If the cookie is absent, fall back to Accept-Language parsing:
   - Header indicates Portuguese (`pt`, `pt-BR`, `pt-br`, etc.) → 302 redirect to `/pt-br/`.
   - Otherwise → render the en feed at `/`.
4. The redirect uses HTTP status 302 (Found), not 301 (Moved Permanently). Google interprets 302 as locale negotiation rather than a permanent move.
5. The response on `/` SSR includes `Vary: Cookie, Accept-Language` to keep CDNs honest about response cacheability.
6. Crawler behavior: bots that do not send Accept-Language (Googlebot, Bingbot in their default crawl mode) stay at `/` and index the en canonical content. Hreflang pairs guide them to the pt-br variant.

The same mechanism applies to the post-feed root only. Post detail pages and About do not auto-redirect — they serve whatever locale matches the URL, with the translation banner UX (preserved from current state per PRD) when content is missing in the requested locale.

## Alternatives Considered

### Alternative 1: Always Accept-Language

- **Description**: Every request to `/` does Accept-Language parsing, redirecting accordingly. No cookie precedence.
- **Pros**: Slightly simpler logic.
- **Cons**: Returning visitors who manually chose `en` get re-redirected to `/pt-br/` on every visit. Caching layers can't cache `/` reliably. Some scrapers spoof Accept-Language and get the wrong content.
- **Why rejected**: Cookie precedence is a one-line addition that solves the returning-visitor and cache problems.

### Alternative 2: Client-side JavaScript redirect

- **Description**: `/` always SSRs en. Client JS detects browser language and calls `window.location.replace('/pt-br/')`.
- **Pros**: No server-side redirect; pure client logic.
- **Cons**: Flash of incorrect content; bots cannot follow JS redirects; worst latency (full page load + JS execute + redirect).
- **Why rejected**: SEO and UX both worse than SSR redirect.

### Alternative 3: Stick with ADR-001 no-redirect

- **Description**: `/` always renders en. Portuguese visitors must use the locale switcher.
- **Pros**: Simplest design. Cleanest cache semantics. No `Vary` header complexity.
- **Cons**: Friction for Portuguese first-time visitors. Manual switcher click required.
- **Why rejected**: Author wants Portuguese browsers to land in their locale automatically; the friction reduction is worth the complexity if pt-br content lands meaningfully.

## Consequences

### Positive

- Portuguese-speaking visitors land at `/pt-br/` on first visit without manual switcher interaction.
- Visitor's explicit locale choice persists via cookie; subsequent visits do not re-redirect against their preference.
- Bots without Accept-Language stay at `/` (en canonical) and index correctly.
- Hreflang `<link rel="alternate">` pairs preserve cross-locale ranking equity.
- 302 status signals locale negotiation to search engines, not a permanent move.

### Negative

- `Vary: Cookie, Accept-Language` on `/` is required for CDN correctness; reduces cache hit ratio at `/`.
- Slightly more SSR cost on `/` than option 3 (header parsing + potential redirect).
- One extra HTTP round-trip for first-time pt-br visitors (small; mitigated after first visit by cookie).

### Risks

- **Bot or crawler spoofs Accept-Language**. Some scrapers send `Accept-Language: pt-BR`. They get redirected. Indexing implication: if Googlebot ever spoofs Accept-Language in a quality crawl, pt-br could become the perceived canonical. Mitigation: hreflang pairs are authoritative; even if a bot lands on `/pt-br/`, the canonical link on that page points to `/pt-br/` correctly.
- **CDN cache poisoning**. Without `Vary` headers respected, a CDN could serve pt-br content to en visitors. Mitigation: explicit `Vary` header; document caching behavior; verify in production via curl-with-headers test post-deploy.
- **Cookie not honored after locale switch**. If the cookie is set but `detectLocaleFromRequest` reads it incorrectly, the user can be redirected against their choice. Mitigation: unit-test `detectLocaleFromRequest` with cookie present, cookie absent + Accept-Language, cookie absent + no Accept-Language.

## Implementation Notes

- The redirect logic lives in the root index route's `beforeLoad` handler (or equivalent SSR-only hook in the `{-$locale}/index.tsx` route from ADR-004).
- The cookie set by `LocaleProvider` must use `Path=/` and a reasonable `Max-Age` (e.g., one year). Confirm the current implementation in `app/lib/locale.tsx` matches.
- Response headers must include `Vary: Cookie, Accept-Language` on `/` regardless of whether a redirect occurs. Implement via TanStack Start's response header mechanism (Nitro `useResponseHeader` or similar).
- Unit tests must cover: (a) cookie=en → no redirect, (b) cookie=pt-br → 302 to /pt-br/, (c) no cookie + Accept-Language: pt → 302, (d) no cookie + Accept-Language: en → no redirect, (e) no cookie + no Accept-Language → no redirect (en default).
- PRD's Non-Goals section explicitly listed "Accept-Language auto-redirect" as out of scope. This ADR formally supersedes that statement; the PRD reader should be guided to this ADR via the ADR-004/ADR-005 cross-references.
