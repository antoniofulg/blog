# ADR-007: Adopt Zod as the Project's Validation Primitive

## Status

Accepted

## Date

2026-05-14

## Context

ADR-001 specifies a typed Zod schema for both the `UIStrings` i18n contract and the About MDX frontmatter. The codebase has zero Zod imports today. All existing runtime validation is done with plain TypeScript functions (e.g., `validateLocaleFn` in `app/routes/$lang/blog.server.ts`, `parseFrontmatterBlock` in `app/db/indexer.ts`). These ad-hoc validators throw on bad input but offer no schema export, no error formatting, and no type-inference reuse.

V1 introduces two new validation surfaces:
- About MDX frontmatter (`title`, `locale`, `links[]`).
- The `UIStrings` module (typed locale-keyed string maps).

The decision is whether to introduce Zod now (a new dependency, a new pattern) or extend the plain-TS validators.

Codebase context:
- TanStack Start `createServerFn().inputValidator(fn)` accepts any function returning the validated value or throwing. A Zod schema's `.parse` fits this signature directly.
- The `posts` indexer's `parseFrontmatterBlock` would benefit from a Zod schema (`postFrontmatterSchema`), but that refactor is out of V1 scope.
- Zod 4.x is the current major version; its bundle size is small (~12 KB minified+gzipped).

## Decision

Adopt Zod as the project's validation primitive. Install `zod` as a runtime dependency.

V1 scope of Zod adoption:
1. `app/lib/i18n/strings.ts` — `uiStringsSchema = z.object({ ... })` validates the `UIStrings` shape. The module's load-time validation calls `uiStringsSchema.parse(strings[locale])` per locale; missing keys throw at import time.
2. `app/lib/mdx/about-schema.ts` (or co-located in `about.server.ts`) — `aboutFrontmatterSchema = z.object({ title: z.string(), locale: z.enum(LOCALES), links: z.array(linkSchema).optional() })`. The About loader calls `.parse(frontmatter)` after `parseFrontmatter`.

Out of V1 scope but enabled by this decision:
- Retroactively replacing `validateLocaleFn` and other inline server-fn validators with `z.enum(LOCALES).parse`.
- Retroactively replacing `parseFrontmatterBlock` validation in the posts indexer with `postFrontmatterSchema`.

Both retrofits are V2 candidates. V1 introduces the pattern in two surfaces and leaves the rest for later.

## Alternatives Considered

### Alternative 1: Plain TS validators (no new dep)

- **Description**: Write `validateAboutFrontmatter(data: unknown): AboutFrontmatter` and `validateUIStrings(strings: unknown): UIStrings` as hand-written type guards or validators that throw on invalid input.
- **Pros**: No new dependency. Mirrors the existing `validateLocaleFn` pattern.
- **Cons**: Schema definition and the TypeScript type live in two places (validator + type). Error messages are bespoke per validator. No schema export for documentation or testing. Hand-written validators are easy to write incorrectly (especially for arrays of objects like `links[]`).
- **Why rejected**: The duplication and error-message bespoke-ness costs more than Zod's bundle. Zod's `.infer` keeps schema and type in sync from one source.

### Alternative 2: Adopt Zod project-wide in V1

- **Description**: Add Zod, then refactor every existing inline validator in `*.server.ts` files to use Zod schemas. Posts indexer, locale validators, server fn input validators all use Zod.
- **Pros**: Consistent validation primitive everywhere; one source of truth for shape.
- **Cons**: Project-wide refactor expands V1 scope significantly; touches files unrelated to the site restructure; risks regressions in unrelated areas.
- **Why rejected**: V1 should introduce Zod where new validation is needed; retroactive refactors are V2 work.

## Consequences

### Positive

- About frontmatter and UIStrings module have schema-driven validation; missing or malformed fields fail loudly at parse/load time.
- Zod's `.infer<typeof schema>` keeps TypeScript types and runtime shape in sync from a single source.
- Future server fn input validators have a standard pattern (`createServerFn().inputValidator((input) => mySchema.parse(input))`).
- Zod error formatting is built-in; we get descriptive validation errors for free.

### Negative

- New runtime dependency (~12 KB minified+gzipped); negligible for SSR but worth noting.
- Mixed validation styles during V1 — Zod for new surfaces, plain validators in legacy code. Mitigation: V2 retrofit candidate.

### Risks

- **Zod major version churn**. Zod 4.x is stable; the API has been backward-compatible for years. Mitigation: pin to a known-good minor version; upgrade deliberately.
- **Server-only schema imports leaking to client bundle**. If `about-schema.ts` is imported from both client and server, the schema runs client-side too. Acceptable for small schemas; Zod itself is isomorphic. Mitigation: server-only validation lives in `.server.ts` files marked in `serverOnlyStubPlugin` if needed.

## Implementation Notes

- Add Zod via `bun add zod` during Phase 3.
- Place schemas adjacent to their consumers:
  - `app/lib/i18n/strings.ts` — `uiStringsSchema` + `type UIStrings = z.infer<typeof uiStringsSchema>`.
  - `app/lib/mdx/about-schema.ts` — `aboutFrontmatterSchema` + `type AboutFrontmatter = z.infer<typeof aboutFrontmatterSchema>`.
- The schemas use `LOCALES` as the source for `z.enum(LOCALES)` so adding a locale to `app/lib/locale.tsx` propagates automatically.
- Loader pattern:
  ```typescript
  const raw = parseFrontmatter(filePath);
  const frontmatter = aboutFrontmatterSchema.parse(raw); // throws on invalid
  ```
- Unit tests cover schema validation success and failure cases for both `aboutFrontmatterSchema` and `uiStringsSchema`.
