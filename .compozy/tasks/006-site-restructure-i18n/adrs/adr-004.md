# ADR-004: Optional Path-Param `{-$locale}` Routing Primitive

## Status

Accepted

## Date

2026-05-14

## Context

V1 needs a default-locale `/` to coexist with `/pt-br/` (Portuguese prefix) while sharing route components. ADR-001 fixes the URL surface but defers the routing primitive to TechSpec.

Three idiomatic TanStack Router options exist:

1. Optional path-param `{-$locale}` — TanStack Router v1.169+ supports optional segments. A single route file at `/{-$locale}` handles `/`, `/<slug>` (with `{-$locale}` resolving empty for default), `/pt-br/`, `/pt-br/<slug>`. The `$lang` param is optional and resolves to `undefined` (interpreted as the default locale) when omitted.
2. Explicit `/` route + reuse `$lang/` subtree — Keep `$lang.tsx` + `$lang/blog.tsx` + `$lang/$slug.tsx`. Add new top-level `/index.tsx` + `/$slug.tsx` routes that re-render the same components, passing `locale="en"` manually. Two physical paths per page; shared components but duplicated route wiring.
3. Promote `$lang.tsx` to handle `/` — Modify `$lang.tsx` so a missing `$lang` param defaults to `"en"`. Requires custom route matcher / catch-all logic.

Codebase context:
- TanStack Router plugin version supports optional path params per official i18n example (`react/i18n-paraglide`).
- `routeTree.gen.ts` is auto-generated by the vite plugin on file changes.
- `$lang.tsx` already validates the locale param against `LOCALES`.

## Decision

Adopt the optional path-param idiom. Restructure routes to use `{-$locale}` segments:

- `/{-$locale}` — locale layout (replaces `$lang.tsx`); resolves param to `DEFAULT_LOCALE` when undefined.
- `/{-$locale}/` (index) — post feed at locale root. Handles `/` (en) and `/pt-br/`.
- `/{-$locale}/$slug` — post detail. Handles `/<slug>` and `/pt-br/<slug>`.
- `/{-$locale}/about` — About page. Handles `/about` and `/pt-br/about`.

The layout component reads `params._locale` (or whatever TanStack exposes for the optional name), falling back to `DEFAULT_LOCALE` from `app/lib/locale.tsx` when undefined. `beforeLoad` validates the resolved locale against `LOCALES`.

Server fns continue to receive the resolved (always-defined) `Locale` value; the optional-param mechanic is purely a routing concern and does not affect data layer or query signatures.

## Alternatives Considered

### Alternative 1: Explicit `/` + reuse `$lang/` subtree

- **Description**: Add `app/routes/index.tsx` and `app/routes/$slug.tsx` that import and re-render the same components used by `/$lang/blog.tsx` and `/$lang/$slug.tsx`, hardcoding `locale="en"`.
- **Pros**: No reliance on optional-param syntax stability; explicit per-URL files.
- **Cons**: Duplicated route wiring; two places to update when adding a new locale-aware page; not idiomatic per TanStack Router i18n examples.
- **Why rejected**: Idiomatic optional-param keeps every locale-aware page in one route file. Duplication scales poorly when adding a third locale or new pages.

### Alternative 2: Promote `$lang.tsx` to handle `/`

- **Description**: Modify `$lang.tsx` to treat an empty param as `DEFAULT_LOCALE`. Add a catch-all that strips the param requirement.
- **Pros**: Most code reuse; routing logic concentrated in one layout.
- **Cons**: Requires custom matcher logic; departs from the file-based router conventions; harder for future contributors to reason about.
- **Why rejected**: The optional-param idiom achieves the same code-reuse benefit without custom matcher code.

## Consequences

### Positive

- One route file per locale-aware page. No duplication for `/` vs `/pt-br/`.
- Matches the TanStack Router i18n example (`react/i18n-paraglide`) — known-good pattern.
- Server fns and queries are unaffected; the optional-param resolution happens in the route layer only.
- Future locales (e.g., `/es/`, `/fr/`) plug in without restructuring route files — they just need entries in `LOCALES` and corresponding content directories.

### Negative

- Requires TanStack Router v1.169+ (already satisfied; current installed version is 1.169.1).
- `routeTree.gen.ts` syntax for optional segments is newer; less community Q&A exists for debugging edge cases.
- Existing `$lang.tsx` and `$lang/` subtree must be migrated (route file renames + parameter access updates).

### Risks

- **Optional-param resolution behavior at `/`**. The exact API for accessing the resolved value (e.g., `params._locale` vs `params['-locale']`) depends on TanStack Router internals. Mitigation: prototype during Phase 2 implementation; fall back to Alternative 1 if the optional-param API proves unreliable for SSR.
- **Hot reload of moved route files**. Renaming `$lang.tsx` to `{-$locale}.tsx` may briefly break dev-server route generation. Mitigation: stop the dev server during the rename, restart, verify `routeTree.gen.ts` regenerates cleanly.

## Implementation Notes

- Phase 2 of the rollout (per ADR-002) owns this routing restructure.
- File renames (preserving git history via `git mv`):
  - `app/routes/$lang.tsx` → `app/routes/{-$locale}.tsx`
  - `app/routes/$lang/blog.tsx` → `app/routes/{-$locale}/index.tsx`
  - `app/routes/$lang/blog.server.ts` → `app/routes/{-$locale}/index.server.ts`
  - `app/routes/$lang/$slug.tsx` → `app/routes/{-$locale}/$slug.tsx`
  - `app/routes/$lang/$slug.server.ts` → `app/routes/{-$locale}/$slug.server.ts`
  - Add `app/routes/{-$locale}/about.tsx` + `about.server.ts` in Phase 3.
- Delete old top-level shims: `index.tsx`, `blog.tsx`, `$slug.tsx`, `about.tsx`.
- Update any imports referencing the old paths.
- Verify hreflang generation correctly references both `/` and `/pt-br/` (no `/en/` prefix).
