# ADR-001: V1 Scope for Site Restructure and Organic Content

## Status

Accepted

## Date

2026-05-13

## Context

The blog is a WIP personal tech site with no production traffic. Several pages render mocked data inline in route files: `tutorials.tsx` and `tutorials.$seriesSlug.tsx` (hardcoded series arrays), `projects.tsx` (hardcoded projects array), `about.tsx` (hardcoded profile, stack, experience, fake social links). Blog post feed lives at `/$lang/blog` while `/` and `/blog` are redirect shims. Header `NAV_LABELS` and footer link arrays are hardcoded inline in component files. Footer links to `/feed.xml`, `/sitemap.xml`, `/robots.txt` which return 404. Newsletter and Search routes are placeholder stubs ("Em breve" disabled form, empty component).

Existing locale infrastructure is well-built but underutilized:
- `app/lib/locale.tsx` exports `Locale = "en" | "pt-br"`, `LOCALES`, `DEFAULT_LOCALE`, `detectLocaleFromRequest`, `LocaleProvider`, `useLocale`.
- `app/routes/$lang.tsx` validates locale param against `LOCALES`.
- `app/routes/$lang/blog.tsx` and `$lang/$slug.tsx` already locale-aware.
- DB `posts` table has `lang` column with unique `(slug, lang)` constraint.
- `app/db/indexer.ts` derives lang from parent directory.
- Content dirs `content/en/` (3 mdx files) and `content/pt-br/` (`.gitkeep` only) exist.

Council debate (4 advisors: pragmatic-engineer, architect-advisor, product-mind, devils-advocate) converged on a scope that locks i18n schema contracts in V1 while deferring full string extraction until Portuguese content actually exists.

## Decision

Accept the following V1 scope:

### Deletions (zero regret)

1. Delete `app/routes/tutorials.tsx`, `tutorials.$seriesSlug.tsx`, `projects.tsx`.
2. Delete `app/components/tutorial-step.tsx`.
3. Delete `app/routes/newsletter.tsx`, `search.tsx` (placeholder stubs).
4. Delete `app/routes/blog.tsx`, `index.tsx`, `$slug.tsx` redirect shims.

### Route restructure

5. Blog post feed renders at `/` for English; `/pt-br/` renders Portuguese feed. Move logic from `$lang/blog.tsx` into `$lang/index.tsx` (or equivalent), with `/` collapsing to the default-locale variant via a TanStack Router optional-param idiom.
6. Post URLs: `/<slug>` (English) and `/pt-br/<slug>` (Portuguese).
7. About: `/about` and `/pt-br/about`. Content sourced from `content/<locale>/about.mdx`.
8. No Accept-Language auto-redirect; `/` always renders English.

### About MDX schema

9. Free body MDX for prose (bio, narrative).
10. Typed Zod frontmatter:
    - Required: `title`, `locale`.
    - Structured optional with rendering consequences: `links[]` (each with `label`, `url`, `kind`).
    - Other display fields nullable with default-empty fallback in component.

### i18n string contract

11. Add `app/lib/i18n/strings.ts` defining `type UIStrings = { ... }` and `const strings: Record<Locale, UIStrings>` plus a Zod schema.
12. Populate `strings` ONLY with keys required for Portuguese post rendering in V1: route titles, locale switcher label, post meta labels (e.g., "Published on", "Reading time"), 404 copy on locale-aware routes.
13. Header nav labels, footer link labels, brand copy, OG meta keys: keep hardcoded in V1; defined in the same `UIStrings` type as TODO comments so the V2 extraction is an additive diff, not a retrofit.

### Cleanup

14. Move `content/en/lorem-ipsum.mdx` to `app/tests/fixtures/lorem-ipsum.mdx`. Update fixture references in tests.
15. Rewrite the two tests that reference `/about` (`public-routes.test.ts:154`, `header.test.ts:149`) to assert locale-aware About rendering.
16. Remove broken footer links to `/feed.xml`, `/sitemap.xml`.
17. Ship `/robots.txt` as a static 5-line route (allow all, point to canonical host).
18. Remove `Tutorials` and `Projects` from header `NAV_LABELS` and footer `navLinks`.

### Out of V1 scope (V2+)

- Full nav/footer/brand UI string extraction.
- `/sitemap.xml` route.
- `/feed.xml` RSS route.
- Newsletter provider integration.
- Search implementation.
- Auto-scaffold pt-br companion post when en post added.

## Alternatives Considered

### Alternative 1: Defer all i18n string extraction to V2

- **Description**: Ship deletions + route restructure + About migration. Keep all UI strings hardcoded. Add i18n module only when first pt-br post lands.
- **Pros**: Smallest V1 surface; matches the "no readers, no infra" principle.
- **Cons**: Format chosen ad-hoc later under time pressure becomes a retrofit cost; "extracted" vs "not extracted" boundary becomes implicit and unowned.
- **Why rejected**: Schema contract is cheap (one file, one type) and pre-deciding it prevents drift across feature branches.

### Alternative 2: Full UI string extraction in V1

- **Description**: Extract every nav label, footer link, 404 copy, OG meta, and brand string into locale modules in V1.
- **Pros**: Single pass; no V2 extension work.
- **Cons**: Speculative work for a locale (`pt-br`) whose readership is zero; opportunity cost vs content pipeline / admin features.
- **Why rejected**: Opportunity cost is concrete; deferred strings can be appended to an already-typed schema without migration.

### Alternative 3: Free body About with no frontmatter schema

- **Description**: About MDX is pure prose; component reads body only.
- **Pros**: Lowest authoring friction; no schema to maintain.
- **Cons**: `pt-br/about.mdx` can silently omit structured rendering fields (links list, etc.) and the section renders empty without indexer-time error.
- **Why rejected**: Locale variants are written months apart; the second-locale author is effectively a different person from a contract perspective. Silent rendering failures are the worst failure mode.

### Alternative 4: Keep `/blog` redirect, add `/pt-br/blog` as alias

- **Description**: Maintain `/blog` and `/pt-br/blog` as canonical, treat `/` as marketing landing.
- **Pros**: No URL churn.
- **Cons**: Dual surface (`/` vs `/blog`) with no clear canonical; landing pages on personal blogs are friction without traffic to justify them.
- **Why rejected**: Post-feed-at-`/` matches the minimalist personal blog pattern winning in 2026 (HackerNoon, Marketer Milk references).

### Alternative 5: Date-based post URLs `/<lang>/<year>/<month>/<day>/<slug>`

- **Description**: WordPress-style date hierarchy in post URLs.
- **Pros**: Freshness signal; archive nav.
- **Cons**: Tech tutorials look stale at 2-year mark even when evergreen; slugs globally unique already so dates add no collision value; 5 path segments are uglier for sharing.
- **Why rejected**: User-confirmed during clarification round; dates show in post header + listing card instead.

## Consequences

### Positive

- Zero mocked data in user-facing pages — Antonio's content credibility uplift, especially in About.
- `/` becomes real content (post feed), not a redirect shim — first-paint UX improves.
- Locale parity foundation laid: every page can serve `en` and `pt-br`.
- DB `lang` column and `$lang` route infrastructure finally exercised end-to-end.
- Typed i18n schema contract prevents V2 retrofit cost.
- `/robots.txt` removes broken-link audit noise.
- 5 routes + 1 component deleted = smaller surface area, fewer paths for dead code to accumulate.

### Negative

- `routeTree.gen.ts` regeneration required after route deletions and moves.
- 2 tests need rewriting.
- 1 fixture file moves location.
- V2 must extend (not migrate) the i18n schema — manageable but requires discipline.

### Risks

- **Default-locale `/` collision**: TanStack Router optional-param idiom (`{-$locale}`) is the recommended primitive but may interact awkwardly with existing `$lang.tsx` layout. Mitigation: prototype during PRD phase; fallback is explicit `/` route plus `/$lang/*` routes with shared component.
- **Indexer Zod failures**: If About frontmatter drifts during pt-br authoring, indexer crashes block deploys. Mitigation: keep frontmatter schema small (only `title` + `locale` required; `links[]` structured but optional with default-empty).
- **SEO churn**: If Google has indexed `/tutorials`, `/projects`, or `/blog`, those become 404s. Mitigation: WIP justification; consider 410 Gone if reach matters later.
- **i18n schema scope drift**: Pressure to extract "just one more string" during V1. Mitigation: ADR-001 strictly defines which keys ship in V1; deviations require a new ADR.
