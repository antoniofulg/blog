# ADR-003: Fold Post Indexing Stabilization into V1 Phase 1

## Status

Accepted

## Date

2026-05-14

## Context

While reviewing the V1 PRD draft, the author observed that the admin dashboard at `/admin/` displayed zero posts despite `.mdx` files existing in `content/en/`. Investigation surfaced a structural defect in the indexing lifecycle:

- `scripts/sync.ts` is the only entry point that bulk-populates `posts` rows from filesystem content via `syncAll(contentDir)`.
- `scripts/watcher.ts` runs a separate watcher process; it reacts to `.mdx` file events but does not perform an initial sync.
- `bun run dev` (the vite dev server) does not trigger either script. Posts in `content/` are invisible to the admin dashboard until a developer manually runs `bun run sync`.
- `.github/workflows/cd.yml` runs `bun run db:migrate` then `docker compose up -d` against the VPS. There is no `bun run sync` step. After every deploy, the `posts` table in production is empty until something else populates it.

The bug predates the multilang/locale work; it was masked while the content directories were empty (only a single `lorem-ipsum.mdx` fixture existed) and exposed when the author opened the admin dashboard after recent content/structure changes.

The bug is small in surface but load-bearing: the V1 PRD assumes admin posts are visible (so the author can publish/unpublish during restructure), assumes Phase 2 routing changes can be verified against indexed content, and assumes Phase 3's MDX migration for About leverages the same indexer pipeline. Shipping V1 on top of an unreliable indexing lifecycle multiplies the risk of phase verification gaps.

## Decision

Fold post indexing stabilization into V1 Phase 1 as a new feature (F9 in the PRD). The scope:

1. **Deploy pipeline** — Insert a `bun run sync` step in `.github/workflows/cd.yml` after `bun run db:migrate` and before `docker compose up -d --no-deps app`. The sync runs against the baked `content/` directory inside the runner image so production `posts` rows reflect the deployed content state on every release.
2. **Dev experience** — Make `bun run dev` perform an initial `syncAll` against `./content` at boot, then start the watcher in the same process (or document that `bun run sync && bun run dev` is the developer workflow if a one-step solution is judged unsafe in the TechSpec).
3. **Admin verification** — Add a quality criterion in PRD Phase 1 success: admin dashboard at `/admin/` renders a non-empty post list (assuming at least one `.mdx` exists in `content/`) after deploy.
4. **Failure observability** — Sync failures during deploy must fail the workflow loudly (no silent swallowing); failures during dev boot log an actionable message and continue to start the server in degraded mode.

Phase 1 remains "slim down" themed because F9 is a stabilization concern that complements the deletion work — both ensure the foundation is honest before restructuring URLs and content in Phases 2 and 3.

## Alternatives Considered

### Alternative 1: Spin off as a separate pre-V1 hotfix

- **Description**: Land the indexer fix as a separate `hotfix/post-indexing` PR before V1 work starts. Keep PRD scope unchanged.
- **Pros**: PRD stays clean; clean separation between bug fix and restructure work; smallest possible PR for the indexer fix.
- **Cons**: Adds branch/PR coordination overhead before V1; loses the "V1 is the moment when the foundation becomes reliable" narrative; risks the hotfix being delayed or forgotten, leaving V1 to inherit the broken state anyway.
- **Why rejected**: User explicitly preferred folding into V1. The fix is small enough that bundling it with Phase 1 deletions does not bloat the diff meaningfully, and the PRD becomes the authoritative record of all V1 stabilization work.

### Alternative 2: V1 Phase 0 (verification gate)

- **Description**: Add a fourth phase before Phase 1, named "Verify foundations." Includes the indexer fix plus a manual smoke test of admin and existing routes.
- **Pros**: Cleanest separation of concerns; explicit verification gate before the restructure begins.
- **Cons**: Adds a fourth phase to a V1 that the user already chose as 3-phase per ADR-002; verification-only phases tend to bloat with checklists; the fix is small enough to be a Phase 1 feature without confusing the rollout shape.
- **Why rejected**: A separate phase is overhead for a single fix; Phase 1's "slim down + stabilize" theme accommodates F9 naturally.

### Alternative 3: Defer fix to V2

- **Description**: Document the bug in PRD Open Questions and leave V1 phases assuming the manual `bun run sync` workaround.
- **Pros**: Smallest V1 scope; no ADR change.
- **Cons**: Phase 1 success criteria become harder to verify (admin shows posts only if developer remembers manual sync); Phase 2 and 3 verification all depend on the same fragile workaround; V2 inherits the same defect.
- **Why rejected**: The fix is too small to defer responsibly; deferring a bug that affects every V1 phase's verifiability is poor scoping discipline.

## Consequences

### Positive

- After Phase 1 ships, deploys reliably populate the `posts` table without manual intervention.
- Admin dashboard becomes verifiable as a Phase 1 success criterion (a stake in the ground for "the foundation is honest").
- Phases 2 and 3 inherit a working indexer pipeline; their routing and content verification become cheaper.
- Dev workflow becomes one-step (`bun run dev`) — onboarding friction reduced.
- The PRD becomes the authoritative scope record; no separate hotfix to track.

### Negative

- Phase 1 diff grows by approximately one workflow file edit (`cd.yml`) and one dev-boot integration point (entry-point script or vite plugin hook).
- The PRD review cycle for V1 extends by one ADR creation and one feature line.
- A failing sync step during deploy now blocks the deploy entirely — previously the workflow could pass with an empty `posts` table. This is a stricter contract and intentional, but it raises the cost of broken content files.

### Risks

- **Sync failure on deploy due to malformed `.mdx` frontmatter**. Mitigation: Phase 1 PR includes a smoke pass of `bun run sync` against current `content/` to verify clean state before merging.
- **Dev boot becomes slower if sync is run at startup**. Mitigation: `syncAll` is bounded by content volume (currently 3 files); acceptable for V1. If volume grows, TechSpec can revisit (e.g., file-mtime caching, async background sync).
- **Watcher and initial-sync race condition**. If the dev server starts the watcher before the initial sync completes, an `.mdx` edit during sync could be missed. Mitigation: sequence the boot — `syncAll` first, then start watcher.

## Implementation Notes

- TechSpec decisions deferred to next pipeline stage (`cy-create-techspec`):
  - Whether dev-time sync runs as a vite plugin hook, a wrapper script, or a TanStack Start server-start hook.
  - Whether deploy sync runs inside the runner container (via `docker exec`) or as a separate workflow step against a sidecar image.
  - Whether failure-mode logging uses the existing structured-JSON pattern from `watcher.server.ts`.
- The `bun run sync` step in `cd.yml` should run against the deployed image so the indexer sees the same `content/` directory the app will serve.
- F9 verification: after Phase 1 ships, log in to `/admin/` against production and confirm post rows render. Confirm at least one row reflects a current `content/` `.mdx` file.
