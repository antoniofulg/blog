# ADR-006: About Page Served from MDX Without DB Persistence

## Status

Accepted

## Date

2026-05-14

## Context

ADR-001 specifies that About migrates to MDX per locale: `content/en/about.mdx`, `content/pt-br/about.mdx`. ADR-001 left storage strategy to TechSpec, listing two candidate alternatives (reuse `posts` table with a `kind` column vs separate `about_content` table).

A third option emerged during TechSpec: do not persist About at all. Read the MDX file directly at request time, render with the existing MDX pipeline, return the HTML. About is a singleton per locale (1 file per locale, at most a handful of fields); there is no list view, no pagination, no published/draft toggle.

Codebase context:
- The MDX pipeline (`app/lib/mdx/parser.server.ts` + `app/lib/mdx/renderer.server.ts`) already exposes `parseFrontmatter(filePath)` and `renderMdx(source)` as synchronous-ish server functions.
- The MDX renderer uses a Shiki singleton highlighter (already warm after first call).
- TanStack Start route loaders can read files server-side; the result is cached per-render via loader memoization.
- The current `posts` table is post-shaped (publishedAt, viewCount, series, seriesPart) and adding About rows would require either nullable fields or a `kind` discriminator.

## Decision

Serve About directly from the MDX file at request time. No DB row, no schema migration, no indexer change for About.

Behavior:
1. Route loader at `/{-$locale}/about` receives the resolved `locale`.
2. Loader calls a new helper `loadAbout(locale: Locale)` that:
   - Resolves the file path `content/<locale>/about.mdx`.
   - Reads the file via `node:fs/promises`.
   - Parses frontmatter with `parseFrontmatter` (existing).
   - Validates frontmatter against a Zod schema (`aboutFrontmatterSchema`, see ADR-007).
   - Compiles MDX body to HTML via `renderMdx`.
   - Returns `{ frontmatter, html, locale, fallbackLocale? }`.
3. If `content/<locale>/about.mdx` does not exist for the requested locale, loader falls back to the default locale (en). The returned object includes `fallbackLocale` so the component can render the existing `TranslationNotice` banner.
4. The About component renders the typed frontmatter fields (title, links) and dangerously sets the compiled MDX body, mirroring the post detail rendering pattern.

The About loader is a normal TanStack Start `createServerFn().handler()` — no auth gate, public read.

## Alternatives Considered

### Alternative 1: Reuse `posts` table with a `kind` column

- **Description**: Add `kind text not null default 'post'` to the `posts` table. About rows have `kind='about'`. Indexer routes by file path (e.g., files matching `content/<locale>/about.mdx` indexed with `kind='about'`).
- **Pros**: Single indexer pipeline; single query layer; matches the existing data model.
- **Cons**: `posts.kind='about'` is semantically awkward; About-specific fields (e.g., the `links[]` array) don't fit the posts schema; nullable post-only fields (publishedAt, viewCount) pollute About queries; migration adds a `kind` column and backfill that V1 does not need.
- **Why rejected**: About is a singleton with different shape; the DB does not add value for fetching one file at one path per locale.

### Alternative 2: Separate `about_content` table

- **Description**: New table with own columns (`locale`, `title`, `body`, `links`, `indexedAt`). Indexer routes About files to this table.
- **Pros**: Clean semantics; About has its own schema.
- **Cons**: A new table for one row per locale (currently 2 rows total) is overhead; requires migration, query layer, indexer routing logic.
- **Why rejected**: Same overhead as Alternative 1 with no offsetting benefit. The DB does not need to store what a static file already represents authoritatively.

## Consequences

### Positive

- Zero schema migration for About. The `posts` table remains post-shaped.
- Zero indexer change for About. F9 indexing pipeline applies only to `posts`.
- About authoring is the simplest possible: edit `content/<locale>/about.mdx`, commit, deploy. No sync step required, no DB row to worry about.
- The TanStack Start loader caches the read+render per-render naturally; subsequent requests in the same SSR pass reuse the result.
- Zod validation at loader time catches frontmatter errors immediately with a clear error.

### Negative

- No DB-level analytics on About (e.g., a `viewCount` field). Not in scope; analytics is a separate concern.
- Each request to `/about` performs a file read + MDX compile. Mitigation: Shiki highlighter is a warm singleton; About body is small; production VPS has filesystem cache.

### Risks

- **File path drift if `content/` directory moves**. The loader hardcodes `content/<locale>/about.mdx`. Mitigation: centralize the content path in a constant exported from `app/lib/mdx/paths.ts` (or similar) so future changes touch one location.
- **MDX compile cost on every request**. For high-traffic sites this matters; for a WIP personal blog, it does not. If About traffic spikes, add a per-locale in-memory cache keyed by file mtime; defer until measured.
- **Fallback-locale rendering**. When `pt-br/about.mdx` does not yet exist, the loader falls back to en and the component must set `lang="en"` on the article wrapper for a11y. Mitigation: unit-test the fallback path.

## Implementation Notes

- New helper: `app/lib/mdx/about.server.ts` exports `loadAbout(locale: Locale): Promise<AboutContent>` where `AboutContent` is the typed result.
- The Zod schema for About frontmatter lives in `app/lib/mdx/about-schema.ts` (or co-located in `about.server.ts`). See ADR-007 for the Zod adoption.
- Route file `app/routes/{-$locale}/about.tsx` calls `loadAbout` via a server fn in `about.server.ts`.
- Hreflang pairs on About route render alongside the existing pattern in `$slug.tsx`.
