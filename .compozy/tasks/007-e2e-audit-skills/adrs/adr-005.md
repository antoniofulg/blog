# ADR-005: Revive app-audit as Phase 4 — supersedes ADR-002 deferral

## Status

Accepted (supersedes the deferral portion of [ADR-002](adr-002.md); does NOT revert the audit-pivot decision in ADR-002 — content-audit remains shipped + canonical for the content-drift surface)

## Date

2026-05-19

## Context

[ADR-002](adr-002.md) pivoted the audit scope from browser-sweep to content-audit and deferred `app-audit` to a V2 contingency. The deferral trigger was explicit: "If `content-audit` produces sustained signal AND a browser-class regression escapes prod" OR "two consecutive zero-actionable-finding runs ⇒ evaluate retirement and pivot." Neither condition has fired — `content-audit` shipped in Phase 3 but has not accumulated enough runs to gate the pivot.

After completing Phases 1-3 and closing 4 review rounds, the user identified a recurring need that neither shipped skill covers: **ad-hoc browser inspection** for console errors, network failures, missing meta tags, hydration warnings, image health, mixed-content warnings, slow responses, a11y violations, and Lighthouse-measured perf/SEO/best-practices regressions across every route × locale × auth-state combination — without writing a Playwright spec per concern.

The shared infrastructure delivered in Phases 1-3 (site-model producer, PGLite test harness, auth fixture, reporter helpers including the `escapeMarkdownCell` safety wrapper from round 2 issue 004, and PR-comment delta logic) absorbs ~40% of the build cost for a third audit-shaped skill (line-counted estimate: ~250 of ~600 LOC genuinely lifted from existing modules). The choice was between (a) shipping `app-audit` in this branch as Phase 4 with the existing infra, or (b) opening a follow-up PRD-008 after the deferral trigger fires later.

A council session re-debated the reversal with 4 advisors (Pragmatic Engineer, Architect, Devil's Advocate, The Thinker). The Thinker reframed the proposal: `app-audit` is NOT a third renderer in the symmetric sense — it is a **fuzzer** that uses the site-model as a *classifier*, not as an oracle. The Architect demanded cross-cutting-surface hardening (SUMMARY.md namespacing + fingerprint typed union) BEFORE the third consumer lands. Devil's Advocate held that the deferral trigger is exactly the forcing function the prior council installed against premature scope expansion. Pragmatic Engineer initially favored extending the branch then partially conceded that Lighthouse-in-V1 introduces unmanageable variance.

After rebuttals, the council converged on a HYBRID recommendation: ship app-audit conditionally (probe gate, hardening first, Lighthouse deferred). The user reviewed the council synthesis and elected to proceed with the ORIGINAL scope unmodified — all 11 categories including Lighthouse CI, 5 tasks not 7, no probe-first gate, no shared-surface hardening as a prerequisite step. User judgment: cost-amortization window + context-freshness outweighs the council's variance + atrophy concerns; reversibility is cheap if app-audit needs to be disabled later.

This ADR documents the user's final accepted scope (original proposal) and records the council's hybrid alternative + Devil's Advocate's deferral case in the Alternatives Considered section for the historical record.

## Decision

Reverse the `app-audit` deferral from ADR-002 unconditionally, and ship `app-audit` as Phase 4 of TASK-0007 inside the existing `TASK-0007/e2e-audit-skills` branch with the original proposed scope:

1. **Skill name**: `app-audit`. **Slash command**: `/app-audit`. **Makefile targets**: `make audit-fe` (alias `make app-audit`) + composite `make audit` running `audit-content` + `audit-fe` sequentially.

2. **CI gate**: informational, NOT merge-blocking (same posture as `content-audit` per ADR-002 rationale).

3. **Coverage matrix**: `routes (from getRouteInventory()) × locales (en + pt-br) × auth-state (anon + admin)` — 28 inspections per run at current site scale. Expected wall-clock: 2-5 min per run with PGLite + Playwright + axe + Lighthouse.

4. **Categories (11)**:
   - `console-error` (blocker) — `page.on("console")` filter on `msg.type() === "error"`.
   - `hydration-mismatch` (blocker) — console filter for "hydration failed" / "did not match" / "Text content does not match".
   - `network-fail` (blocker for 5xx / major for 4xx) — `page.on("requestfailed")` + `response.status() >= 400`.
   - `broken-image` (major) — `page.locator("img").evaluateAll(...)` checks `naturalWidth === 0`.
   - `missing-meta` (major) — assert `<title>`, `<meta name="description">`, `<meta property="og:title">`, `<meta property="og:image">`, `<link rel="canonical">`, `<meta name="viewport">` present + non-empty.
   - `mixed-content` (major) — console filter for "Mixed Content" warnings.
   - `slow-response` (minor) — `performance.timing` / `performance.now()` heuristic; flag if first-paint > 1.5s.
   - `a11y-violation` (major) — `new AxeBuilder({ page }).withTags(["wcag2a", "wcag2aa", "wcag22aa"]).analyze()` via `@axe-core/playwright` direct call (NOT delegated to the `a11y-testing` skill, to keep the audit deterministic and avoid skill-invocation overhead per route).
   - `seo-score-drop` (minor) — Lighthouse CI integration; flag if `categories.seo.score < 0.9`.
   - `perf-budget-breach` (minor) — Lighthouse perf score < 0.8.
   - `best-practices-fail` (minor) — Lighthouse best-practices score < 0.9.

5. **Output** (mirror content-audit shape):
   - Per-run report: `docs/_reports/app-audit-YYYY-MM-DD.md` (gitignored, uploaded as GHA artifact).
   - Committed summary row: `docs/audits/SUMMARY.md` extended with a `Type` column (`content` / `app`). Existing content-audit rows backfilled with `content` value during Phase 4.
   - PR comment: delta-only via `peter-evans/create-or-update-comment@v4` with a separate fingerprint marker (`<!-- audit-fingerprint:app:blocker=X major=Y -->`) so app-audit and content-audit comments coexist on the same PR without colliding.
   - Lighthouse HTML report: GHA artifact alongside the markdown report.

6. **Architecture — reuse Phase 1-3 infrastructure**:
   - Site-model producer (`app/lib/site-model.server.ts`) — no changes; `RouteEntry` shape stays unchanged.
   - PGLite harness (`tests/e2e/db.ts`) — reused.
   - Auth fixture (`tests/e2e/fixtures/auth.ts`) — reused for admin-session walks.
   - New modules: `app/lib/app-audit/browser-sweep.server.ts`, `checks.server.ts`, `a11y-adapter.server.ts`, `lighthouse.server.ts`, `reporter.server.ts`. All five added to `vite.config.ts:serverOnlyStubPlugin`.
   - `escapeMarkdownCell` helper shared from content-audit's reporter (no duplication).
   - CLI: `scripts/audit-fe.ts` with `--trigger`, `--routes`, `--lighthouse` flags; emits `[audit-counts]` machine-readable line (mirrors round 2 issue 003 pattern).
   - Workflow: `.github/workflows/app-audit.yml` — `workflow_dispatch` + paths-filtered PR trigger on `app/routes/**`, `app/components/**`, `app/lib/**`, `app/db/schema.ts`.
   - SKILL.md: `.agents/skills/app-audit/SKILL.md` + `.claude/skills/app-audit` symlink + `.claude/commands/app-audit.md` slash alias.
   - Rule file: `.agents/rules/fe-audit.md` (severity scheme, category definitions, abort condition, finding row format, triage workflow).

7. **`audit-fe` exit code**: 1 if any `blocker` finding, 0 otherwise (mirrors `audit-content`).

8. **Abort condition**: 3 consecutive zero-actionable-finding runs (vs content-audit's 2-run rule). Higher threshold reflects app-audit's fuzzer nature — naturally surfaces more findings per run.

9. **Phase 4 structure: 5 tasks** (task_16 through task_20):
   - task_16: Browser sweep core (`browser-sweep.server.ts` with console + network + meta + image + mixed-content + hydration + slow-response listeners)
   - task_17: Checks orchestrator + reporter (`checks.server.ts` + `reporter.server.ts`; `SUMMARY.md` Type-column refactor)
   - task_18: CLI + workflow + Makefile targets (`scripts/audit-fe.ts` + `.github/workflows/app-audit.yml` + `make audit-fe` + composite `make audit`)
   - task_19: A11y adapter + Lighthouse integration (`a11y-adapter.server.ts` + `lighthouse.server.ts` + `@axe-core/playwright` + `@lhci/cli` devDeps)
   - task_20: SKILL.md + slash alias + `.agents/rules/fe-audit.md` + AGENTS updates

## Alternatives Considered

### Alternative 1: Council's hybrid recommendation (probe gate + Lighthouse deferred + hardening first)

- **Description**: Ship app-audit conditionally. Phase 4a delivers shared-surface hardening (SUMMARY.md namespaced into H2 sections, fingerprint typed union module) BEFORE renderer code. Phase 4b runs a manual probe against the current site-model and emits a Vitest assertion reporting `%` of synthetic findings the model could not classify. If `<30%` actionable, Phase 4 aborts. Phase 4c adds sibling `app/lib/site-model/expectations.ts` ONLY if probe justifies. Lighthouse CI excluded from V1; deferred to PRD-009 with a `≤3-point variance` baseline gate.
- **Pros**:
  - Probe-first sequence converts the deferral trigger from speculation into discovery-gated work.
  - Lighthouse deferral removes the highest-risk noise vector (±10 perf-point swings on shared CI runners).
  - Shared-surface hardening prevents accidental debt accumulation at N=3 consumers; SUMMARY.md namespacing scales to N=5+ audit types cleanly.
  - Sibling expectations table keeps `RouteEntry` boundary tight (Architect's "extend, don't bend" principle).
  - Fuzzer framing makes the build smarter, not bigger (Thinker's reframe).
- **Cons**:
  - 7 tasks instead of 5 (+0.5 day).
  - Conditional task (expectations.ts) adds branching complexity in the task graph.
  - Mid-branch abort signal could cause commit-revert; emotionally costly.
  - Lighthouse perf/SEO/best-practices remain uncovered until follow-up PRD; CDN cache + OG image regressions still escape.
- **Why rejected**: User judged the council's noise + atrophy concerns as manageable post-merge (skills are independently invokable; disabling the workflow is one PR). Cost-amortization window + context-freshness outweighed the council's risk pricing. User explicit accepts Lighthouse variance risk + skips probe gate, with the understanding that reversibility is the safety net.

### Alternative 2: Honor ADR-002 deferral as-is

- **Description**: Decline the reversal. Ship the current branch with e2e-coverage + content-audit only. Wait for ADR-002's trigger ("two consecutive zero-actionable content-audit runs" OR "≥1 prod bug escapes both skills") to fire before opening PRD-008 for `app-audit`.
- **Pros**: Honors the prior council's deliberate forcing function. Avoids speculative scope growth. PR stays at 9.6K lines, 4 review rounds, ready to merge.
- **Cons**: User has identified a real need that neither shipped skill covers. Waiting on the trigger means accepting prod bugs as the discovery mechanism. The shared-infra cost-amortization window closes once the branch merges (~40% savings becomes ~5-day standalone PRD-008 cost).
- **Why rejected**: Devil's Advocate's strongest argument, but the user accepted the cost-amortization claim and chose proactive coverage over reactive pivot.

### Alternative 3: Open PRD-008 as a separate branch immediately

- **Description**: Close TASK-0007 at round-4 state, merge to main, open new branch `TASK-0008/app-audit` with its own full planning cycle (idea → PRD → techspec → tasks).
- **Pros**: Cleanest historical record. ADR-002 deferral preserved. Each branch reviews in isolation.
- **Cons**: Re-runs ~1h of planning bootstrap (re-reading site-model patterns, re-deciding shared-surface integration). Forces another full 4-round review cycle on smaller surface. Total effort ~5 days vs ~3 days inline.
- **Why rejected**: Process-purity tax exceeds the architectural benefit. ADR-005 documents the reversal; the planning artifacts already cite Phase 4 explicitly; no historical-record loss.

### Alternative 4: Original minus Lighthouse only

- **Description**: Adopt all 11 categories except `seo-score-drop`, `perf-budget-breach`, `best-practices-fail`. Skip Lighthouse + `@lhci/cli` devDep. Keep probe + hardening sequence skipped (no Thinker reframe applied). 5 tasks; ~2.5 days.
- **Pros**: Removes the highest-variance category set. Cheaper devDep footprint.
- **Cons**: Loses perf + SEO + best-practices signal that the user explicitly wanted ("SEO issues"). Half-measure.
- **Why rejected**: User explicitly wants the full proposal.

## Consequences

### Positive

- **40% shared-infra reuse confirmed** (line-counted: ~250 of ~600 LOC lifted from existing modules: PGLite proxy ~100%, auth fixture ~100%, `escapeMarkdownCell` ~100%, PR-comment workflow template ~100%, reporter scaffolding ~80%).
- **Single coherent "quality skills" trio delivery** — e2e-coverage + content-audit + app-audit ship together in one PR to main.
- **Context-freshness preserved** — build proceeds with the shared infra still warm in the developer's head.
- **Composite `make audit` makes co-atrophy visible** at the command level (if app-audit atrophies, content-audit's noise will surface in the same command output).
- **User has stated need fulfilled** without writing a Playwright spec per concern.

### Negative

- **Branch grows from 9.6K to ~13K LOC** (+35%). Reviewer surface expands; likely 1-2 additional review rounds.
- **Lighthouse variance accepted as known risk** — perf-score swings of ±10 points between identical runs on shared CI runners may cause PR-comment fatigue. Mitigation deferred to operational discipline post-merge (treat scores as advisory, not gate signals).
- **Atrophy of three skills** is a real risk that the council flagged twice (ADR-001 Devil's Advocate, ADR-005 Devil's Advocate). Mitigated by composite `make audit` + abort condition (3 zero-finding runs) + reversibility (skills independently disable-able).
- **Site-model boundary** could come under pressure from app-audit's needs (e.g., per-route expected-presence metadata). Mitigated by discipline: ADR-005 explicitly states `RouteEntry` shape stays unchanged; any audit-specific metadata lives in a separate module if added later.
- **No probe gate** means the build proceeds on the assumption that current site-model + 28 inspections will produce actionable signal. If post-merge data shows >70% of findings are noise, the abort condition (3 zero-actionable runs) is the only formal recovery mechanism.

### Risks

- **PR-comment fatigue from Lighthouse variance**: most likely failure mode within 60 days. *Mitigation*: per-comment severity counts (delta-only) keep the comment from re-posting unless signal changes; if Lighthouse-driven false alarms exceed a threshold, the `--lighthouse` flag in CLI allows disabling that category set without touching the rest of the skill.
- **Site-model boundary creep**: app-audit may surface findings that the current `RouteEntry` cannot classify, creating pressure to add fields. *Mitigation*: ADR-005 commits to "no widening of `RouteEntry`"; any new audit metadata grows as a sibling module in a future ADR.
- **SUMMARY.md `Type` column drift**: existing content-audit rows must be backfilled with `Type: content`. *Mitigation*: task_17 explicitly includes a migration step; Vitest test asserts every row has a `Type` value before the suite passes.
- **Fingerprint regex collision**: app-audit's `audit-fingerprint:app:` and content-audit's `audit-fingerprint:content:` markers must NEVER cross-match. *Mitigation*: task_18's workflow file uses a literal-string grep (`grep -F "audit-fingerprint:app:"`) on the previous comment body, not a regex with backreferences.
- **Devil's Advocate's 6-month post-mortem scenario**: "We shipped 3 audit skills in one PR. e2e + content-audit are useful. app-audit produces 14 findings per run, 12 are stale Lighthouse noise. PR comments are ignored. Skill was disabled in workflow file 6 weeks after merge." *Mitigation*: documented retirement decision tracker in `## App Audit` SUMMARY section; the abort condition (3 zero-actionable runs) is the formal escape valve.

## Implementation Notes

- **ADR ordering**: this ADR cites ADR-001 / ADR-002 / ADR-003 / ADR-004 explicitly. ADR-002 is partially superseded (deferral lifted with the user's chosen scope); the audit-pivot decision in ADR-002 stands.
- **PRD-007 updates**: add Phase 4 to "Phased Rollout Plan"; extend "Core Features" with F11-F15 (browser sweep core, checks + reporter, CLI + workflow, a11y + Lighthouse adapters, skill docs); extend "Success Metrics" with app-audit KPIs (≥1 actionable finding in 50% of runs over first 90 days); update "Non-Goals" to remove the explicit browser-sweep exclusion; add Open Questions for Lighthouse variance management + abort-condition tuning.
- **TechSpec-007 updates**: add "Phase 4 — app-audit" section with the 5-task build order; new Core Interfaces (`AppAuditFinding`, `BrowserSweepResult`, `AuditType` union, `LighthouseScores`); updated Impact Analysis table for the new modules.
- **`_tasks.md`**: append rows for task_16 through task_20; rebuild dependency graph.
- **Branch grows on the same branch** (`TASK-0007/e2e-audit-skills`); no fork.
- **DevDep additions**: `@axe-core/playwright`, `@lhci/cli`, `lighthouse` (transitive of `@lhci/cli`).
- **Lighthouse runner mode**: headless Chrome via Playwright's bundled Chromium; same binary as e2e specs (cache hit on CI).
- **`audit-fe` exit code 1 on blocker** ensures `workflow_dispatch` failure is explicit; the GHA workflow itself does NOT block merge (informational gate per ADR-002 posture).
- **`a11y-testing` skill remains canonical for component-level a11y work** — app-audit's a11y check is route-level via direct `AxeBuilder` calls; the two coexist without overlap.
