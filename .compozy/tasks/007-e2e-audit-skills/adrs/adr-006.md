# ADR-006: TechSpec implementation primitives for Phase 4 (app-audit)

## Status

Accepted

## Date

2026-05-19

## Context

[ADR-005](adr-005.md) reactivated `app-audit` as Phase 4 with the original 11-category scope. The TechSpec phase surfaces four implementation primitives that downstream tasks (task_16 through task_20) depend on. Each primitive has multiple defensible options; locking them here prevents drift during execution.

The four primitives are:

1. **Lighthouse default state** in CLI + workflow — runs always / opt-in / hybrid local-vs-CI.
2. **Lighthouse Chromium binary** — share Playwright's bundled Chromium or let `@lhci/cli` download its own.
3. **Browser sweep error handling** — fail-fast on first probe error or per-route try/catch + emit a synthetic finding.
4. **PR comment fingerprint scope** — counts-only or counts + category/finding hash.

Each was settled in a dedicated TechSpec clarification question with the user. This ADR consolidates them so downstream task files reference a single primitives module.

## Decision

### 1. Lighthouse default: ON locally, OFF in CI

`scripts/audit-fe.ts` defaults `--lighthouse` to `true` when invoked locally (no `CI` env var) and to `false` when invoked from a CI environment (`CI === "true"`). The CLI accepts `--lighthouse` / `--no-lighthouse` flags as explicit overrides. The GitHub Actions workflow `.github/workflows/app-audit.yml` exposes a `workflow_dispatch` input `lighthouse` (default `"false"`) that overrides CI's off-default when the developer wants a manual perf/SEO check from the GitHub UI.

Rationale: signal at dev time (developer is investigating before promoting a draft), no variance noise on every PR (Lighthouse perf scores swing ±10 points between identical runs on shared GHA runners, producing PR-comment fatigue). Local-on captures the user's stated headline workflow (pre-publish content sanity check across the entire site). CI-off accepts ADR-005's variance-tolerance trade-off explicitly.

```typescript
// scripts/audit-fe.ts
function parseLighthouseFlag(args: string[]): boolean {
  if (args.includes("--no-lighthouse")) return false;
  if (args.includes("--lighthouse")) return true;
  return process.env.CI !== "true"; // default ON locally, OFF in CI
}
```

### 2. Lighthouse Chromium: share Playwright's bundled binary

`@lhci/cli` is configured to use the Chromium binary already cached by `bunx playwright install chromium` (Phase 1 setup). The Lighthouse runner module passes the resolved Playwright Chromium path via `chromePath` in the lhci config:

```typescript
// app/lib/app-audit/lighthouse.server.ts
import { chromium } from "@playwright/test";

async function runLighthouse(url: string): Promise<LighthouseScores> {
  const chromePath = chromium.executablePath();
  // pass chromePath to lhci runner via collect.settings.chromePath
  // ...
}
```

Rationale: single cache hit on CI (one ~150 MB binary instead of two); faster cold start (~30 s saved per workflow run); no Puppeteer/Lighthouse Chrome download dependency. Trade-off accepted: tight coupling between Playwright's Chromium version and Lighthouse's expected version; if Playwright's Chromium ever predates Lighthouse's minimum supported version, fall back to lhci's own download.

### 3. Browser sweep error handling: per-route try/catch + `sweep-error` finding

Each route × locale × auth-state inspection in `app/lib/app-audit/browser-sweep.server.ts` runs inside a `try/catch` block. On any thrown error (Playwright timeout, browser crash, fixture failure, network unreachable), the catch handler emits a synthetic finding and continues with the next route. The whole-run report always completes; partial failures appear in the finding list with the route identified.

```typescript
// app/lib/app-audit/browser-sweep.server.ts
type SweepError = {
  category: "sweep-error";
  severity: "major";
  filePath: string;   // route path
  message: string;    // error.message
  detail: { stack?: string };
};

async function sweepRoute(route: RouteEntry, ...): Promise<Finding[]> {
  try {
    return await runProbes(route);
  } catch (err) {
    return [{
      category: "sweep-error",
      severity: "major",
      filePath: route.path,
      message: err instanceof Error ? err.message : String(err),
      detail: { stack: err instanceof Error ? err.stack ?? "" : "" },
    }];
  }
}
```

Rationale: V1's 28 inspections must not be aborted by a single transient probe failure. Surfacing the failure as a major finding keeps signal honest (developer sees the route name + error message) without breaking the entire sweep. The `sweep-error` category is added to the V1 set as a 12th category; it does NOT count toward the abort condition (3 zero-actionable-finding runs) because it represents probe-infrastructure failure, not site quality.

### 4. PR comment fingerprint: counts-only

The fingerprint marker embedded in the PR comment body is:

```html
<!-- audit-fingerprint:app:blocker=X major=Y -->
```

Suppression fires when both `blocker` and `major` counts match the previous comment on the same PR. Minor count changes are ignored to prevent low-severity-driven re-posts.

Rationale: matches content-audit's existing pattern exactly (round 2 issue 003 + round 4 issue 002 in earlier rounds). Simplest possible delta logic. Trade-off accepted: if categories churn (e.g. 3 broken-link findings replaced by 3 missing-meta findings, same counts), no re-post fires. Deemed acceptable for V1; revisit if evidence shows category-churn drift goes undetected for >2 weeks.

## Alternatives Considered

### Lighthouse default alternatives

1. **Default OFF + opt-in via flag** — CLI/workflow always require explicit enable.
   - **Cons**: developer must remember to enable for pre-publish flow; loses headline user story's "one command sees everything".
   - **Rejected**: degrades local UX.
2. **Default ON + opt-out via flag** — always runs unless disabled.
   - **Cons**: CI noise + variance fatigue exactly as ADR-005 warned.
   - **Rejected**: defeats the variance-mitigation that ADR-005 acknowledged.
3. **Default OFF locally + ON in CI** — inverted hybrid.
   - **Cons**: developer never sees Lighthouse output during dev; CI eats the variance hit on every PR.
   - **Rejected**: wrong direction on every axis.

### Lighthouse Chromium alternatives

1. **Let `@lhci/cli` download its own Chrome** — simplest config.
   - **Pros**: no path-passing complexity.
   - **Cons**: 2 browser binaries on CI (~300 MB), longer cold start.
   - **Rejected**: cost outweighs config simplicity at the V1 scale (Phase 1 already cached Chromium).
2. **System Chrome with fallback** — auto-detect.
   - **Cons**: inconsistent across dev environments (Chrome version drift); CI reproducibility risk.
   - **Rejected**: ADR-004 already prefers explicit dependency pinning.

### Browser sweep error alternatives

1. **Fail-fast** — first error aborts whole run.
   - **Cons**: loses partial results; one flaky route blocks the entire report.
   - **Rejected**: too brittle for V1's 28-inspection matrix.
2. **Per-route try/catch + silent stderr log** — no finding emitted.
   - **Cons**: hides per-route failures from the markdown report.
   - **Rejected**: bad triage UX; user has no way to discover failures from the report.
3. **Sliding-window threshold** — only emit finding after N consecutive failures.
   - **Cons**: complex; absorbs single-route failures silently.
   - **Rejected**: premature optimization without evidence of transient noise volume.

### Fingerprint scope alternatives

1. **Counts + category hash** — `sha256(sorted-category-names).slice(0,8)` appended.
   - **Pros**: catches category churn at same counts.
   - **Cons**: more complex; diverges from content-audit's existing pattern.
   - **Rejected**: pattern consistency wins at V1; revisit if drift discovered.
2. **Counts + top-finding hash** — hash of top-severity message + filePath.
   - **Pros**: single-finding fidelity.
   - **Cons**: misses changes in lower-severity findings.
   - **Rejected**: same drawback as counts-only without the simplicity.
3. **Counts + Lighthouse score buckets** — encode `perf=80-89` ranges.
   - **Pros**: most signal-fidelity for perf variance.
   - **Cons**: Lighthouse-specific complexity in a fingerprint that should stay simple.
   - **Rejected**: variance-management belongs in the follow-up PRD, not in V1 fingerprint.

## Consequences

### Positive

- **Local-on Lighthouse** matches user's headline story (pre-publish full-site audit) without forcing CI variance pain.
- **CI-off Lighthouse** is overridable via `workflow_dispatch` input for explicit perf investigations, preserving the capability without the per-PR cost.
- **Shared Chromium binary** saves CI cache + cold-start time + disk space.
- **Per-route try/catch + `sweep-error`** keeps the 28-inspection matrix robust; partial failures are observable, not silent.
- **Counts-only fingerprint** matches content-audit's pattern exactly; round 2/4 fixes apply unchanged.
- All four primitives are reversible decisions: flag flipping, config tweaks, no schema or external API commitments.

### Negative

- **Local-on default** means a developer running `make audit-fe` for the first time may not realize Lighthouse is enabled and may be surprised by the longer wall-clock (~2-5 min vs ~30-60 s without). Mitigation: `audit-fe` summary line logs whether Lighthouse ran.
- **Shared Chromium coupling** means a Playwright minor-version bump could break Lighthouse if the bundled Chromium ages out of lhci's supported range. Mitigation: pin both packages explicitly; document fallback in `.agents/rules/fe-audit.md`.
- **`sweep-error` finding** is a 12th category that wasn't in the original ADR-005 scope. PRD-007's F11 still says "11 categories"; the TechSpec adds `sweep-error` as a probe-infrastructure category (not counted toward abort condition).
- **Counts-only fingerprint** misses category-churn signal. Mitigation: documented in `.agents/rules/fe-audit.md`; revisit if drift observed.

### Risks

- **Lighthouse version drift with Playwright Chromium**: likelihood medium. *Mitigation*: pin `@lhci/cli` + `lighthouse` + `@playwright/test` versions explicitly; CI runs a compatibility check (`bunx lighthouse --version && bunx playwright --version`) before each app-audit run.
- **`sweep-error` category masking deeper issues**: if a probe consistently fails on the same route, the `sweep-error` finding hides the underlying assertion-level finding (e.g., the route really has a hydration mismatch that's making the probe time out). *Mitigation*: triage workflow in `.agents/rules/fe-audit.md` instructs the developer to investigate `sweep-error` findings locally with `bunx playwright test --headed --debug` to surface the root cause.
- **CI `workflow_dispatch` input parsing brittleness**: GitHub Actions inputs are strings; "false" vs `false` vs `"false"` vs unset all behave differently. *Mitigation*: workflow uses `${{ github.event.inputs.lighthouse == 'true' }}` (literal string equality) with default `"false"`.
- **Fingerprint regex collision still possible** at the workflow level if a future audit type prefix matches `app:` substring. *Mitigation*: use `grep -F "<!-- audit-fingerprint:app:"` (literal-string match, not regex) consistent with round 2 issue 003 fix carried through Phase 3.

## Implementation Notes

- **`scripts/audit-fe.ts` flag-parsing helper** lives at module top; covered by Vitest unit tests for all 4 input combinations (`--lighthouse`, `--no-lighthouse`, `CI=true`, `CI` unset).
- **`app/lib/app-audit/lighthouse.server.ts`** imports `chromium.executablePath()` from `@playwright/test`. Module added to `vite.config.ts:serverOnlyStubPlugin`.
- **`app/lib/app-audit/browser-sweep.server.ts`** exports `SweepError` type alongside `BrowserSweepResult`; `Finding` type union in `checks.server.ts` extended to include `sweep-error` category.
- **`.github/workflows/app-audit.yml`** declares `workflow_dispatch.inputs.lighthouse` (type: `choice`, options: `["false", "true"]`, default `"false"`). Workflow step conditional: `if: ${{ inputs.lighthouse == 'true' }}`.
- **PR-comment fingerprint module** at `tests/e2e/audit-fingerprint.ts` exports `AuditType = "content" | "app"` union and `formatFingerprint(type, counts): string`. Content-audit's existing fingerprint code refactored to import from this module in Phase 4 (small co-located change with task_17's reporter delivery).
- **`SUMMARY.md` `Type` column migration** runs once before the first app-audit append. Migration logic lives in `app/lib/app-audit/reporter.server.ts:initSummary()`; existing content-audit rows backfilled with `Type: content` via in-place sed-style edit gated by detecting absence of the column header.
