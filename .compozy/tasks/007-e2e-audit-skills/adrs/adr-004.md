# ADR-004: TechSpec implementation primitives

## Status

Accepted

## Date

2026-05-18

## Context

The PRD (PRD-007) and prior ADRs (001-003) settled architecture, scope, and delivery. The TechSpec phase surfaces four interlocking implementation primitives that downstream tasks depend on. Choosing wrong here causes rework in every spec file, the auth fixture, the audit pipeline, and the CI script.

The four primitives are:

1. **Site-model module shape** — how `e2e-coverage` and `content-audit` consume route + post knowledge.
2. **PGLite lifecycle** — how many ephemeral PGLite instances exist during an e2e run and how they map to Playwright workers.
3. **Auth fixture pattern** — how admin-session specs obtain an authenticated browser context.
4. **Content-audit MDX link parsing** — how broken-link checks traverse MDX content.

Each was settled in dedicated TechSpec clarification questions with the user. This ADR consolidates them so downstream task files cite a single architectural-primitives reference.

## Decision

### 1. Site-model: TypeScript module with runtime walker

`app/lib/site-model.server.ts` exports both types and functions. Skills and tests call the functions on demand; no JSON manifest is committed.

```typescript
export type RouteAuthLevel = "public" | "admin";
export type RouteEntry = {
  path: string;          // e.g. "/", "/pt-br/", "/admin", "/pt-br/:slug"
  locale: Locale | null; // null for routes outside the {-$locale} group (e.g., /login)
  auth: RouteAuthLevel;
  expectedStatus: 200 | 302 | 401 | 404;
  intent: string;        // "blog home", "admin dashboard", etc.
};

export type PostEntry = {
  slug: string;
  lang: Locale;
  filePath: string;      // relative to repo root
  frontmatter: PostFrontmatter;
  isPublished: boolean;
  hasTwin: boolean;      // does the counterpart-locale post exist?
};

export async function getRouteInventory(): Promise<RouteEntry[]>;
export async function getPostInventory(): Promise<PostEntry[]>;
```

`getRouteInventory()` walks `app/routes/**/*.tsx` excluding `__root.tsx` and `routeTree.gen.ts`, returning hard-coded inventory entries (one entry per file, matched by glob to a static metadata map). The static map is the source of truth for `auth` / `intent` / `expectedStatus`. A Vitest drift test asserts every file under `app/routes/**/*.tsx` matches an entry and fails on mismatch.

`getPostInventory()` walks `app/content/posts/**/*.mdx` filesystem + cross-references the `posts` table via `db` import. Returns the union with `isPublished` from DB and `hasTwin` computed.

### 2. PGLite lifecycle: per-suite singleton with `workers: 1`

`playwright.config.ts` sets `workers: 1` (serial execution). Playwright `globalSetup` creates one PGLite instance, runs `drizzle-kit push` (programmatic API) for both `app/db/schema.ts` and `app/db/auth-schema.ts`, then seeds the e2e admin user via Better Auth's `auth.api.signUpEmail`. `globalTeardown` calls `pglite.close()`.

The seeded user uses credentials from `E2E_ADMIN_EMAIL` / `E2E_ADMIN_PASSWORD` env vars; locally these default to `e2e@test.local` / `e2e-test-password`. CI gets them from GitHub Secrets.

The dev server boots against this PGLite instance by setting `DATABASE_URL` to the PGLite connection URI (e.g., `postgres://memory`) before spawning the preview server. PGLite's TCP shim makes this transparent to `postgres-js`.

### 3. Auth fixture: Playwright auth setup project + storageState

`playwright.config.ts` declares two projects:

```typescript
projects: [
  {
    name: "setup",
    testMatch: /.*\.setup\.ts/,  // matches tests/e2e/auth.setup.ts
  },
  {
    name: "chromium",
    use: { ...devices["Desktop Chrome"], storageState: ".auth/admin.json" },
    dependencies: ["setup"],     // runs after setup
  },
];
```

`tests/e2e/auth.setup.ts` is a single test that performs the UI login round-trip once and saves storageState to `tests/e2e/.auth/admin.json`. Every subsequent spec inherits the storageState via the project config. Specs that need an anonymous session use `test.use({ storageState: { cookies: [], origins: [] } })` at the top of the file (or a per-test override).

Auth fixture helper at `tests/e2e/fixtures/auth.ts` exports `type AuthedFixture = { page: Page; userId: string }` plus a `freshLogin()` helper for the rare case a spec needs a non-cached session.

### 4. MDX link parsing: mdast (remark) full parse

`content-audit` uses `unified` + `remark-parse` + `remark-mdx` to build the AST of each MDX file, then walks for:

- `link` nodes (markdown `[text](url)`).
- `mdxJsxFlowElement` and `mdxJsxTextElement` nodes whose `name` is `a` or `Link`, extracting the `href` attribute value.

Internal links (starting with `/` or matching the locale-prefix pattern) are validated against the slug set returned by `getPostInventory()`. External links (`http://`, `https://`, `mailto:`, fragment-only `#anchor`) are not validated.

## Alternatives Considered

### Site-model alternatives

1. **Generated JSON manifest** — `bun run inventory:gen` writes a committed `site-model.json`; skills and tests read the JSON.
   - **Pros**: deterministic snapshot; no walk at runtime.
   - **Cons**: manual regeneration step; staleness risk; one more file in PR diffs.
   - **Rejected**: extra workflow step + committed artifact for solo dev = friction without benefit.
2. **Hybrid (types + cached JSON)** — TypeScript walker writes JSON cache when filesystem diverges.
   - **Pros**: caches expensive walks.
   - **Cons**: cache invalidation logic; ~15 routes / ~50 posts make walks <100 ms anyway.
   - **Rejected**: YAGNI.
3. **Inline in each skill (no shared module)** — each skill walks the filesystem independently.
   - **Cons**: re-introduces the duplication ADR-001 eliminated.
   - **Rejected**: defeats the architectural-pattern win.

### PGLite lifecycle alternatives

1. **Per-worker** — fixture creates per-worker PGLite, parallel workers allowed.
   - **Pros**: parallel spec execution.
   - **Cons**: complex fixture lifecycle; only pays off at >5 specs; PGLite single-client limit makes intra-worker parallelism impossible anyway.
   - **Rejected**: V1 has 3 specs; serial is fine.
2. **Per-test** — each test creates+destroys.
   - **Pros**: maximum isolation.
   - **Cons**: ~2-3 s overhead per test; total e2e ≥30 s just for DB lifecycle.
   - **Rejected**: too slow for the 8-minute PR cycle target.
3. **Shared dev DB (docker-compose Postgres)** — reuse existing dev Postgres.
   - **Pros**: faster cold start; closest to prod.
   - **Cons**: dev DB state coupling; manual cleanup risk; conflicts with dev workflow.
   - **Rejected**: defeats ephemeral-DB requirement; ADR-001 documented PGLite-only.

### Auth fixture alternatives

1. **Per-spec login via `authenticatedPage()` fixture** — every admin spec logs in fresh.
   - **Pros**: max isolation; tests that mutate session state are safe.
   - **Cons**: ~2 s per spec login overhead × 2-3 admin specs = wasted CI time.
   - **Rejected**: storageState pattern is Playwright's canonical, and our specs don't mutate session.
2. **Hybrid (storageState default + per-spec opt-out)** — `freshAuthenticatedPage()` available.
   - **Pros**: flexibility.
   - **Cons**: two fixture APIs; ADR-001 mandates narrow surface.
   - **Rejected**: YAGNI; `freshLogin()` helper exists for the rare case without being a competing fixture.
3. **API-level session injection** — fixture calls Better Auth sign-in endpoint, injects cookie.
   - **Pros**: fastest (~50 ms vs ~2 s UI login).
   - **Cons**: UI login path loses coverage; `auth-flow.spec.ts` becomes the only spec that exercises real login UI.
   - **Rejected**: UI login round-trip is a tested capability (per `auth-flow.spec.ts`), not infrastructure to bypass.

### MDX parsing alternatives

1. **Regex on raw markdown** — match `[text](path)` patterns only.
   - **Pros**: ~10x faster (10 ms vs 100 ms per file); zero deps.
   - **Cons**: misses `<Link href="">` or `<a href="">` in JSX blocks; documented constraint required.
   - **Rejected**: silent miss is worse than slow correct. Total parse time <2 s for whole tree is acceptable.
2. **Strip JSX, then regex** — preprocess to remove JSX, regex on remainder.
   - **Cons**: links inside JSX blocks become invisible; brittle stripper.
   - **Rejected**: same correctness risk as pure regex.
3. **External link-check tool (markdown-link-check, lychee)** — shell out.
   - **Pros**: off-the-shelf; covers external URLs too.
   - **Cons**: extra dependency; external URL checks add noise (rate limits, 4xx on archived links); harder to integrate with our slug-set validation.
   - **Rejected**: not aligned with content-audit's targeted internal-link scope.

## Consequences

### Positive

- **One canonical site-model module** — both skills + drift test consume identical types. Refactoring is grep-and-rename.
- **PGLite singleton + workers=1** is the simplest viable lifecycle for V1's 3-spec suite. ~2 s startup amortized over all specs; ~$0 infra.
- **Playwright canonical auth pattern** keeps the auth fixture file small (~30 LOC) and uses framework-native primitives — minimum custom code to maintain.
- **mdast parsing** is robust against future content authors using JSX `<Link>` components in posts (TanStack Router patterns may bleed into MDX).
- All four primitives are reversible decisions (none baked into the schema or external API).

### Negative

- **Site-model static map** (file path → metadata) requires manual update when a route is added; drift test catches this but adds a CI failure mode that needs explanation in `.agents/rules/testing.md`.
- **Workers=1** means e2e can't speed up via parallelization. Acceptable for V1 (3 specs × ~30 s each = 90 s); revisit if suite >5 specs.
- **storageState file** at `tests/e2e/.auth/admin.json` must be gitignored — accidental commit leaks the e2e session token. Mitigation: explicit `.gitignore` entry + ESLint/Biome rule to flag JSON files under `.auth/`.
- **remark + unified deps**: adds `unified`, `remark-parse`, `remark-mdx`, `unist-util-visit` to devDependencies (~200 KB unpacked combined). Acceptable; already pulled transitively by `@mdx-js/mdx`.

### Risks

- **PGLite version skew** — if PGLite ships a breaking change (e.g., updated Postgres major), the e2e schema push might fail. *Mitigation*: pin `@electric-sql/pglite` in `package.json`; upgrades go through PR like other deps.
- **storageState invalidation** — if the seeded user's session expires mid-suite (Better Auth default ~7 days), specs lose authentication. *Mitigation*: setup project re-runs at suite start; storageState is regenerated each CI run.
- **mdast version churn** — `remark-mdx` API has changed between v2 and v3. *Mitigation*: pin to a known-good version; update only with the test suite green.
- **Drift test false positives** — if a route is intentionally not part of the site model (e.g., dev-only diagnostic route), drift test fails. *Mitigation*: opt-out entry in the static map with `expectedStatus: null` to mark "documented exclusion".

## Implementation Notes

- `app/lib/site-model.server.ts` is stubbed in `vite.config.ts:serverOnlyStubPlugin` for client bundle exclusion (matches the pattern for `#/db/client`, `#/lib/auth`, etc.).
- PGLite is loaded via `@electric-sql/pglite`. Drizzle integration uses `drizzle-orm/pglite` driver (separate from the production `drizzle-orm/postgres-js` driver in `app/db/client.ts`).
- `tests/e2e/db.ts` exports a `createTestDb()` function that returns a `{ db: PgliteDb, connectionString: string }` tuple. The connection string is passed via env to the preview server.
- `tests/e2e/auth.setup.ts` location matches Playwright's documented pattern; project dependency is enforced by `playwright.config.ts`.
- mdast parsing module at `app/lib/content-audit/link-parser.server.ts` (lives in app/lib because it's reusable for `cy-create-tasks` future audit extensions); content-audit skill imports it.
- The drift test lives at `app/tests/site-model.test.ts` alongside existing Vitest files.
