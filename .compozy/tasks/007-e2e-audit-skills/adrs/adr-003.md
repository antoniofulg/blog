# ADR-003: PRD scope and phased delivery model

## Status

Accepted

## Date

2026-05-18

## Context

[ADR-001](adr-001.md) and [ADR-002](adr-002.md) settled the architecture (shared site-model + 2 renderers) and the audit pivot (browser-sweep → content-audit). Three downstream decisions remained open at PRD time:

1. **PRD scoping**: should the paired skills be documented in a single PRD or split into separate PRDs (one per skill, one per phase)?
2. **Delivery model**: should the V1 ship as a single PR or as phased PRs that each merge to main independently?
3. **CI failure handling**: what is the developer's expected workflow when e2e fails on a PR — strict block, escape hatch, auto-retry, quarantine?

These are not architectural decisions (those are settled) but they are scope-shape decisions that affect task decomposition by `cy-create-tasks`, the structure of the PRD itself, and the day-to-day developer experience.

A fourth decision — how audit findings reach the developer (PR comment vs file-only vs chat-only) — is an acceptance criterion within the PRD body and does not require its own ADR.

## Decision

**Single PRD covering all 3 phases, delivered as 3 sequential PRs, with strict-block CI gating that combines auto-retry-once + `@flaky` annotation with 48 h SLA.**

Concretely:

1. **PRD scope**: one PRD at `.compozy/tasks/007-e2e-audit-skills/_prd.md` covers both skills (`e2e-coverage` + `content-audit`) and all of V1. Single source of truth for goals, user stories, success metrics, and phased rollout. `cy-create-tasks` produces 3 task files under the same directory (one per phase).

2. **Phased delivery — 3 PRs**:
   - **Phase 1 (foundation + smoke)**: site-model module, auth fixture, Playwright bootstrap + config, `auth-flow.spec.ts`, e2e CI job wired and PR-blocking. Single PR on branch `TASK-0007/e2e-foundation` (or a sub-branch convention). Ships as a working regression gate even with just one spec.
   - **Phase 2 (cluster specs)**: `admin-write.spec.ts` and `public-read.spec.ts` added. Single PR on branch `TASK-0007/e2e-specs`. Builds on Phase 1's foundation; no foundation changes.
   - **Phase 3 (content-audit)**: `content-audit` SKILL.md, MDX walker, frontmatter/translation/link/alt/series checks, GH Action with change-trigger + `workflow_dispatch`, PR-comment integration, `docs/audits/SUMMARY.md` append. Single PR on branch `TASK-0007/content-audit`.

3. **CI failure handling — Strict block + `@flaky` SLA + auto-retry-once**:
   - Playwright runs every spec; CI retries the entire failing test once (`retries: process.env.CI ? 1 : 0` in `playwright.config.ts`).
   - If both attempts fail, the e2e job is red and the PR cannot merge.
   - A spec suspected of flakiness can be tagged with `@flaky` (Playwright tag) plus an ISO-date comment marking when the annotation was added.
   - A CI step (lint-level script) scans for `@flaky`, `.skip`, and `.todo` annotations and fails the build if any annotation's recorded date is older than 48 h. This forces resolution within the window rather than indefinite quarantine.
   - No manual `bypass-e2e` label exists; bypassing the gate requires editing CI or removing the annotation, both of which are visible in PR review.

## Alternatives Considered

### Alternative 1: Two parallel PRDs (PRD-007 e2e-coverage + PRD-008 content-audit)

- **Description**: Split paired skills into separate PRDs with separate task trees. Decouples lifecycles entirely.
- **Pros**: Each PRD can be reprioritized or paused independently. Smaller per-document scope.
- **Cons**: Duplicates context — both PRDs would describe the shared site-model module (introduced in ADR-001 specifically to be shared). The compozy task tree fragments (`007-e2e-audit-skills` becomes two unrelated directories). Reintroduces the framing ADR-002's pivot collapsed (audit and e2e as separate initiatives rather than paired renderers).
- **Why rejected**: Violates the council's "paired delivery on shared site-model" recommendation. The shared module is the load-bearing architectural choice; splitting PRDs hides that.

### Alternative 2: Minimal-scope PRD (Phase 1 only)

- **Description**: PRD-007 scopes ONLY foundation + `auth-flow.spec.ts`. Phases 2 and 3 become follow-on PRDs after MVP ships and the pipeline is proven.
- **Pros**: Smallest commitment. Defers content-audit and the remaining 2 specs until evidence accumulates. Easy first ship.
- **Cons**: Loses the "paired delivery" benefit — the site-model is over-engineered for one consumer. Forces re-planning twice (PRD-008 for specs, PRD-009 for content-audit). High risk of scope drift between PRDs as months pass. Idea's V2 abort condition no longer fits cleanly.
- **Why rejected**: The shared-model architecture only earns its keep with multiple consumers; deferring all of them defeats the design.

### Alternative 3: Single phase / single PR

- **Description**: Build everything (site-model + fixture + 3 specs + content-audit skill + CI + PR-comment integration) on one branch and merge as a single PR.
- **Pros**: Simplest tracking. Single review. No partial states on main.
- **Cons**: PR diff size becomes unreviewable (3-5 days of work in one diff). Long-lived branch invites merge conflicts with content commits. No early signal — the foundation can't catch its first real regression until the entire stack lands.
- **Why rejected**: Pragmatic Engineer's argument from ADR-001 council ("debugging weekend at 11pm Sunday") applies double here. Phased delivery preserves the safety net incrementally.

### Alternative 4: Strict block, no escape hatch (no `@flaky`, no auto-retry)

- **Description**: Red blocks merge always. No quarantine annotation accepted. No automatic retry.
- **Pros**: Maximum discipline. Every flake gets attention immediately.
- **Cons**: For a solo dev, transient flakes (network blip on CI runner, cold-start PGLite contention) become merge-stoppers with no relief. Pragmatic Engineer's "first false positive is also first merge-blocked emergency hotfix" warning from ADR-001 applies.
- **Why rejected**: Auto-retry-once meaningfully dampens transient noise without hiding persistent failures. `@flaky` with SLA preserves the discipline (annotations expire); the combination is strictly better than pure-strict.

### Alternative 5: Strict block + auto-retry only (no `@flaky` annotation)

- **Description**: Retry once. If both attempts fail, block. No quarantine mechanism.
- **Pros**: Simplest mental model. No annotation discipline to maintain.
- **Cons**: Provides no way to track "this spec is known-flaky, fix scheduled." Persistent flakes either get fixed immediately or get .skip'd silently (the worst outcome — no visibility into coverage gaps).
- **Why rejected**: `@flaky` + 48 h SLA gives the developer a tracked, time-bounded escape valve without sacrificing visibility.

### Alternative 6: Manual `bypass-e2e` PR label

- **Description**: Red blocks by default; PR can be merged with a `bypass-e2e` label + justification comment.
- **Pros**: Audit trail in GH. Allows emergency merges.
- **Cons**: Bypass becomes habitual after the third use. The escape valve degrades into the default behavior. Solo dev has no second reviewer to gate label usage.
- **Why rejected**: Trust-based escape valves don't survive solo-dev usage patterns. Annotation-based with expiry is enforceable.

## Consequences

### Positive

- **Single PRD = single source of truth** for the paired-skill initiative. Goals, user stories, and success metrics are stated once and reused across phases.
- **3 PRs / 3 phases** keeps each merge reviewable (short-lived branches, ~1-2 days each), surfaces problems early (Phase 1 ships the gate with one spec — first red catch happens in week one, not week three).
- **`cy-create-tasks` produces a clean task tree** with 3 task files in the same directory; phase ordering is explicit.
- **Strict-block + retry + SLA** is the empirically calibrated balance: auto-retry catches ~50 % of transients, `@flaky` annotation lets the dev acknowledge known-flaky specs without silently disabling them, SLA forces resolution before quarantine becomes permanent.
- **No manual bypass label** means CI red is always meaningful; review attention stays on the actual failure.

### Negative

- **3 PR overhead** means 3 sets of CI runs, 3 changelog entries, 3 GHC image builds. Trivially small cost but real.
- **`@flaky` annotation discipline** requires a CI lint script that checks annotation dates — new infra to maintain (small, ~30 lines of shell or TS).
- **Single PRD will be long** (~3-4 printed pages) covering all 3 phases. Document gravity may favor Phase 1 over Phase 3.
- **48 h SLA** is aggressive for a solo dev with unpredictable hours. Mitigation: the SLA is on the annotation date, which the dev sets — so a long weekend means the annotation goes up Sunday night, not Friday.

### Risks

- **Phase ordering drift**: Phase 2 or Phase 3 could start before Phase 1 ships if the dev gets ahead of themselves. *Mitigation*: PRD's phased rollout section enforces "success criteria per phase" gates; `cy-create-tasks` outputs phase files with explicit "depends on" links.
- **Single PRD outgrows readability**: as content-audit's scope evolves, the PRD becomes harder to keep coherent. *Mitigation*: ADRs absorb scope changes (ADR-002 already did this once); the PRD body stays high-level.
- **`@flaky` SLA gets ignored**: dev never deletes the annotation; CI lint script fires; dev disables the script. *Mitigation*: the script lives in the same PR as Phase 1 (foundation), with a comment explaining why it exists. Disabling it leaves a visible diff.
- **PR-comment integration adds noise**: every content-audit run on a PR with content changes comments on the PR; if findings are routine (translation-gap for a planned-but-not-yet-translated post), the comments become wallpaper. *Mitigation*: PR comment includes severity counts; if `blocker` count is 0 and `major` count is unchanged from last run, comment is suppressed (delta-only commenting).

## Implementation Notes

- PRD location: `.compozy/tasks/007-e2e-audit-skills/_prd.md`.
- Task files: produced by `cy-create-tasks`, one per phase, under same directory.
- Branch naming: `TASK-0007/e2e-foundation`, `TASK-0007/e2e-specs`, `TASK-0007/content-audit`. The Ruleset's `TASK-XXXX/slug` pattern accepts longer slugs.
- CI lint script for `@flaky`/`.skip`/`.todo` date enforcement: lives at `scripts/lint-test-annotations.ts`, wired into the `lint` matrix job in `.github/workflows/ci.yml`.
- PR comment posting: standard `actions/github-script@v7` or `peter-evans/create-or-update-comment@v4` action; logic in `.github/workflows/content-audit.yml` (new workflow).
- `playwright.config.ts` retries setting: `retries: process.env.CI ? 1 : 0`.
- Foundation phase blocks Phase 2 and Phase 3; explicit dependency in task files.
