# ADR-001: V1 scope and architecture for e2e-coverage + app-audit skill pair

## Status

Accepted

## Date

2026-05-18

## Context

The blog repo currently has Vitest unit/integration coverage (31 files under `app/tests/`) but zero end-to-end browser coverage. As features compound, two gaps emerge:

1. **Regression risk** — login, admin publish toggle, and locale-prefixed public reads have no executable contract that exercises them through a real browser.
2. **Latent quality drift** — console errors, broken meta, hydration mismatches, slow responses, and a11y violations accumulate silently between manual checks.

Two new agent skills are proposed (`e2e-coverage`, `app-audit`). Both must respect the existing file-layout invariants: `AGENTS.md` is canonical, `.agents/skills/<name>/SKILL.md` with `.claude/skills/<name>` symlink, rule files under `.agents/rules/`.

The council debate surfaced four tensions: spec count for V1 (1 vs 4), spec naming (URL clusters vs capability targets), skill boundary (two distinct skills vs one site-model with two renderers), and audit trigger (calendar vs change-driven). A reframe by The Thinker collapsed several of these by treating the route inventory as the single producer that both skills render differently.

## Decision

Ship a paired skill set built on a shared route-inventory module, delivered sequentially with PR-blocking CI from day one:

1. **One shared site model**: a typed route inventory (path, locale, auth requirement, expected status, intent) lives in `app/lib/site-model.server.ts` and is the canonical source both skills read. Adding a new route means adding an inventory entry in the same commit.

2. **Two SKILL.md renderers**: `.agents/skills/e2e-coverage/SKILL.md` (executable Playwright specs) and `.agents/skills/app-audit/SKILL.md` (markdown narrative report). Neither duplicates route knowledge; both consume the inventory.

3. **V1 spec set is 3 capability-targeted specs**, not 4 URL-cluster specs:
   - `tests/e2e/auth-flow.spec.ts` — login round-trip, session expiry, logout.
   - `tests/e2e/admin-write.spec.ts` — publish toggle, preview, admin dashboard guard.
   - `tests/e2e/public-read.spec.ts` — post render in both locales, locale switcher, 404 path.

4. **Auth fixture is an owned contract**: `tests/e2e/fixtures/auth.ts` exports a typed `authenticatedPage()` helper with explicit cleanup. `app-audit` imports it on day one as a peer consumer, not as ambient state.

5. **PGLite for ephemeral DB**: `drizzle-kit push` against an in-process PGLite instance + Better Auth seed. Real-Postgres nightly job is documented but not built; trigger to build it = first Drizzle migration using a Postgres-only feature (extensions, RLS, partial indexes).

6. **PR-blocking CI from day one**: no `continue-on-error` warm-up window. Flaky specs land with a `@flaky` Playwright annotation and a 48-hour SLA to fix or quarantine; CI rejects any `.skip`/`.todo` older than that window.

7. **Audit trigger**: `workflow_dispatch` (manual) + change-triggered (`paths` filter on `app/routes/**` and `app/db/**`). Per-run report at `docs/_reports/audit-YYYY-MM-DD.md` (gitignored). Committed signal at `docs/audits/SUMMARY.md` — one row appended per run with date, severity counts, top finding.

8. **a11y delegation**: `app-audit` invokes the existing `a11y-testing` skill (axe-core via `AxeBuilder({page}).withTags(['wcag2a','wcag2aa','wcag22aa']).analyze()`) on each inventoried route. No reimplementation.

9. **Boot mode**: `bun run build && bun preview` for both skills. Dev mode is not supported initially.

10. **Packaging**: SKILL.md is canonical for each skill; slash-command aliases (`/e2e-coverage`, `/app-audit`) are thin wrappers that invoke the corresponding SKILL.md.

## Alternatives Considered

### Alternative 1: Aggressive 4-spec paired V1

- **Description**: Bootstrap Playwright + ship `auth-login.spec.ts`, `admin-posts.spec.ts`, `blog-public.spec.ts`, `blog-locale.spec.ts` simultaneously with `app-audit` reusing the auth fixture day-one.
- **Pros**: Full critical-path coverage immediately. Audit launches with no dependency lag.
- **Cons**: Quadruples weekend-debug surface area when the auth fixture flakes against PGLite. Bakes today's URL topology into spec names (PRD-006 just restructured the site once).
- **Why rejected**: Council consensus (Pragmatic + Architect partial concession) — start smaller, lock in interface decisions, layer specs incrementally.

### Alternative 2: Bootstrap-only / single smoke spec V1

- **Description**: Install Playwright, scaffold config + auth fixture, ship one `login.spec.ts` only. Defer all other specs and the audit skill entirely.
- **Pros**: Minimum debug surface. Fastest to green. Validates the pipeline before scaling.
- **Cons**: Leaves the audit/e2e duality unrealized — the shared site model never materializes. Two interface decisions (fixture contract, capability naming) get postponed and retrofitted under pressure.
- **Why rejected**: Architect held firm that the contract must be decided day one regardless of spec count. The Thinker's site-model reframe loses its leverage if only one renderer exists.

### Alternative 3: e2e-only — kill audit until proven necessary

- **Description**: Ship `e2e-coverage` alone. Defer `app-audit` until evidence (3+ months) shows the sweep catches what e2e misses. Devil's Advocate's position.
- **Pros**: Single forcing function (PR gate) prevents the audit-as-theater failure mode. Solo dev attention concentrated on one moving piece.
- **Cons**: Loses the change-triggered narrative report on every route/migration change. Misses the architectural benefit of a shared inventory (audit would be retrofitted, not co-designed). CDN/OG-class regressions remain invisible.
- **Why rejected**: The Thinker's reframe survives Devil's Advocate's atrophy critique (single producer prevents orphaned skill). Change-triggered audit reverses the no-forcing-function critique. Kept as the V1 abort condition instead: if audit produces zero actionable findings in two consecutive runs, retire it.

### Alternative 4: One combined skill (`site-quality`) with both modes

- **Description**: A single SKILL.md that runs either in `e2e` mode or `audit` mode, sharing all infrastructure internally.
- **Pros**: Tightest coupling = no fixture-contract risk. Smallest surface area to maintain.
- **Cons**: Violates Skills 2.0 composition pattern (one skill = one thing). Slash-command alias becomes ambiguous (`/site-quality --mode=e2e`). Auto-trigger description must cover both intents, which weakens both.
- **Why rejected**: Council preferred two renderers feeding off the shared inventory — same coupling benefit without conflating intents in the prompt surface.

### Alternative 5: Capability-named specs only (no URL clusters ever)

- **Description**: Architect's pure position — every spec is named for a capability (`authenticated-session.spec.ts`, `editorial-write.spec.ts`, `localized-public-read.spec.ts`).
- **Pros**: Survives any future site restructure with zero file renames. Architecturally pristine.
- **Cons**: For a 15-route blog with only 3 V1 specs, the abstraction layer adds cognitive cost without proportional payoff. Pragmatic notes that grep-and-replace across 3 files is an afternoon.
- **Why rejected (partially)**: Adopted in spirit — V1 names ARE capability-shaped (`auth-flow`, `admin-write`, `public-read`) rather than route-shaped (`login`, `admin-posts`, `blog-locale`). Future specs follow the same naming convention; no further abstraction layer is introduced.

## Consequences

### Positive

- **Single producer / zero duplication**: route inventory in `app/lib/site-model.server.ts` is the only place route knowledge lives. Both skills are projections of the same artifact.
- **Atrophy resistance**: each renderer has a forcing function — e2e via PR gate, audit via change-trigger. Neither degrades silently because both consume the same upstream.
- **Capability naming** survives the next site restructure with no spec renames.
- **PGLite-only** keeps CI cold start under 90s; testcontainers complexity deferred until justified.
- **Auth fixture contract** prevents the audit skill from becoming a second-class consumer that retrofits to whatever e2e shipped.
- **Committed `docs/audits/SUMMARY.md`** gives a paper trail without diff noise; full reports stay gitignored.

### Negative

- **Sequential delivery** means the audit skill ships ~1 sprint behind e2e — only 1-2 weeks but Devil's Advocate's atrophy critique technically applies during the gap.
- **Inventory module is new infra** — `app/lib/site-model.server.ts` is a non-trivial module that must stay in sync with `app/routes/`. Manual sync risk.
- **PGLite ≠ prod PG** — schema drift risk if a future feature uses pgvector, partial indexes, RLS, etc. Mitigation depends on the nightly real-PG job being built in time.
- **PR-blocking from day one** with no warm-up window means the first false positive blocks a merge until fixed; Pragmatic warned about this trade-off and the council overruled.

### Risks

- **Inventory drift** — `app/lib/site-model.server.ts` falls out of sync with `app/routes/` if developers forget to update both. *Mitigation*: a Vitest test that walks `app/routes/**/*.tsx` and asserts every route has a matching inventory entry; fails CI if drift detected.
- **PGLite extension blindspot** — green CI hides a prod failure when a migration uses Postgres-only features. *Mitigation*: documented playbook in `.agents/rules/testing.md`; trigger condition (first use of pg-only features) wired to a checklist in the migration PR template.
- **Audit-as-theater** — change-triggered runs accumulate in `docs/audits/SUMMARY.md` but blockers never get fixed. *Mitigation*: abort condition baked in — if two consecutive audit runs produce zero actionable findings, retire the skill rather than maintaining dead infra.
- **Spec flake creep** — the 48h SLA for `@flaky`-tagged specs gets ignored. *Mitigation*: CI explicitly rejects `.skip`/`.todo` annotations older than 48 hours via a timestamp comment.
- **Fixture contract creep** — the auth fixture grows to support both consumers and becomes a god-object. *Mitigation*: type the export surface narrowly (`type AuthedFixture = { page: Page; userId: string; cleanup: () => Promise<void> }`); changes require ADR.

## Implementation Notes

- Build order: `app/lib/site-model.server.ts` first → `tests/e2e/fixtures/auth.ts` → `auth-flow.spec.ts` (proves the stack) → remaining 2 specs → `app-audit` skill consumes the inventory + fixture.
- Rule files to create: `.agents/rules/testing.md` (selector hierarchy, wait strategy, naming, tags), `.agents/rules/audit.md` (output location, severity, coverage matrix, a11y delegation).
- `AGENTS.md` updates: File Structure block (add `tests/e2e/`, `docs/_reports/`, `docs/audits/`); Skill Map (rows for E2E + Audit); Rules list (add testing + audit pointers).
- `.agents/rules/cicd.md` updates: e2e on PR (PR-blocking); audit on `workflow_dispatch` + paths-filtered change-trigger; required secrets `E2E_ADMIN_EMAIL`, `E2E_ADMIN_PASSWORD` for seeded test user.
- `.agents/rules/auth.md` updates: anti-pattern — e2e MUST use seeded test user, never real admin; never commit `storageState.json`.
- `.gitignore`: `tests/e2e/.auth/`, `tests/e2e/storageState.json`, `test-results/`, `playwright-report/`, `docs/_reports/audit-*.md`.
- `package.json` scripts: `test:e2e`, `test:e2e:ui`, `test:e2e:debug`, `audit`.
- Skill packaging: SKILL.md at `.agents/skills/<name>/SKILL.md`, symlink at `.claude/skills/<name> -> ../../.agents/skills/<name>`. Slash-command aliases as thin wrappers.
