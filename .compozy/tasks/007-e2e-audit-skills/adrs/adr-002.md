# ADR-002: Pivot audit skill from browser-sweep to content-audit

## Status

Accepted (supersedes the `app-audit` portion of [ADR-001](adr-001.md))

## Date

2026-05-18

## Context

[ADR-001](adr-001.md) scoped two skills: `e2e-coverage` (Playwright spec runner) and `app-audit` (browser-sweep producing a markdown report — console errors, network failures, broken links, slow responses, missing meta, hydration mismatches, a11y delegated to the existing `a11y-testing` skill).

After ADR-001 was accepted, the opportunity scan (Product Strategist phase of `cy-idea-factory`) surfaced a higher-leverage adjacent direction: the blog's actual high-failure surface is **content drift**, not app-code regressions. PRD-006 (site restructure i18n) just landed, introducing en + pt-br locale prefixes. The dominant failure modes that escape current Vitest coverage are content-shaped:

- An English post is added with no Portuguese translation (or vice versa) — silently appears only in one locale.
- An internal markdown link references a slug that no longer exists or never did.
- Frontmatter typos (wrong `lang` value, missing `publishedAt`, malformed `series`/`seriesPart`).
- Series gaps (part 3 published with part 2 unindexed).
- Image references missing alt text — invisible to readers and accessibility scans.

App-audit would catch console errors and hydration warnings — real but lower-frequency on a 15-route blog with stable component code. Content-audit catches the things that actually break content sites: missing translations, broken cross-references, malformed frontmatter, series consistency.

The Devil's Advocate critique in ADR-001's council session is consistent with this pivot: "the last 4 prod issues were CDN cache + OG image regressions that neither tool would've caught" — that critique is even stronger against app-audit than against content-audit, because content-audit's failure surface is concrete and frequently exercised (every new post commit).

## Decision

Replace the `app-audit` skill scope with a `content-audit` skill. `e2e-coverage` remains exactly as ADR-001 specifies.

`content-audit` SKILL.md scope:

1. **Walk `app/content/posts/**/*.mdx`** and the indexer's published posts table.

2. **Validate frontmatter** against the schema defined in `app/db/schema.ts` ($inferInsert) and the parsing rules in `app/lib/mdx/parser.server.ts`. Required fields: `slug`, `lang`, `title`, `description`, `publishedAt`, `isPublished`. Optional: `category`, `series`, `seriesPart`, `draft`.

3. **Detect translation gaps** — for each `slug` present in one locale, warn if its counterpart in the other locale (en ↔ pt-br) does not exist. Severity: `major`.

4. **Verify internal links** — parse `[text](relative-path)` in MDX bodies; for each non-external link, resolve against the inventoried slug set; flag missing targets. Severity: `blocker` if link is in published post, `minor` if in draft.

5. **Image alt-text presence** — every `![alt](src)` must have non-empty `alt`. Severity: `major` (a11y impact).

6. **Series consistency** — for each `series`, verify `seriesPart` values are contiguous starting at 1 and have no gaps. Severity: `minor`.

7. **Output**:
   - Per-run report: `docs/_reports/content-audit-YYYY-MM-DD.md` (gitignored).
   - Committed summary row: `docs/audits/SUMMARY.md` (append-only, one row per run with date, severity counts, top finding).

8. **Trigger**: `workflow_dispatch` + change-triggered via GitHub Actions `paths` filter on `app/content/posts/**` and `app/db/schema.ts`.

9. **No browser sweep, no Playwright session, no a11y delegation**. Content-audit operates on the filesystem + DB indexer state. The `a11y-testing` skill remains available for component/page-level testing and is not invoked from content-audit (different surface).

10. **Reuses ADR-001's shared site-model pattern**: the route inventory in `app/lib/site-model.server.ts` is extended with a `posts` inventory section that content-audit consumes, alongside `app/content/posts/**/*.mdx` as the source-of-truth filesystem walk.

## Alternatives Considered

### Alternative 1: Keep app-audit as scoped in ADR-001

- **Description**: Ship the browser-sweep audit per ADR-001 (console errors, network failures, broken links, slow responses, missing meta, hydration mismatches, a11y delegation).
- **Pros**: Council-validated. Catches browser-runtime failure modes content-audit cannot detect.
- **Cons**: Lower-frequency surface (component code is stable). Browser sweep across 2 sessions × 2 locales × ~15 routes = expensive (60+ axe invocations per audit run, ~3-5min runtime). Devil's Advocate predicted atrophy.
- **Why rejected**: Misaligned with actual failure mode for a content-driven blog post-PRD-006.

### Alternative 2: Ship both (browser-sweep app-audit AND content-audit as siblings)

- **Description**: Two audit SKILL.md files, both consuming the shared inventory; both append to `docs/audits/SUMMARY.md`.
- **Pros**: Maximum coverage. No surface goes unchecked.
- **Cons**: Doubles audit surface area to maintain. Two SKILL.md prompts compete for the same trigger lexicon ("audit the site" — which skill fires?). Doubles report noise.
- **Why rejected**: Premature. If content-audit produces sustained signal but reveals app-audit-shaped misses, app-audit can be added as ADR-003. Sibling launch violates the council's atrophy mitigation (split attention).

### Alternative 3: Subsume content checks into e2e-coverage

- **Description**: Add a `content-validation.spec.ts` to e2e-coverage that asserts translation parity, link integrity, etc. No separate skill.
- **Pros**: Fewer skills to maintain. Single PR-blocking gate covers everything.
- **Cons**: Content checks aren't browser-shaped — they're filesystem + DB queries. Running them inside Playwright requires booting a browser + dev server for assertions that don't need them. Wrong substrate; wrong forcing function (content-audit doesn't need PR-blocking, since content commits are non-code).
- **Why rejected**: Substrate mismatch. e2e-coverage is for browser-runtime behavior; content-audit is for static + DB-indexed content state.

## Consequences

### Positive

- **Higher-signal audit surface** — content drift is the dominant post-i18n failure mode; this targets it directly.
- **No PGLite required for audit** — content-audit operates on filesystem + production-shaped indexer state; no test DB needed.
- **Simpler to build** — ~2-3 days vs ~3-5 for browser-sweep audit. No two-session orchestration.
- **No skill-trigger collision** — `audit content` and `audit site` are no longer ambiguous; only one audit skill exists.
- **Catches the i18n failure mode PRD-006 introduced** — translation parity is a first-class check.
- **No a11y delegation needed** — the `a11y-testing` skill remains the canonical entrypoint for accessibility checks and is not blurred by content-audit.

### Negative

- **Browser-runtime regressions go uncaught by audit** — console errors, hydration mismatches, missing OG meta, broken external links, slow responses are not detected until e2e fires or production reports them.
- **CDN/cache-layer issues remain invisible** — same blind spot Devil's Advocate flagged; still unaddressed but not worsened.
- **ADR-001 history bifurcates** — readers must follow both ADRs to reconstruct the audit scope. Documentation overhead.

### Risks

- **Translation parity false positives** — short-form posts (e.g., link drops) may intentionally exist in one locale. Mitigation: opt-out via `frontmatter.translatedOnly: false` or `noTranslation: true`.
- **Internal link false positives** — anchored fragments (`/posts/foo#section`) must be parsed without false-flagging the fragment. Mitigation: strip fragments before slug lookup; never validate fragment existence.
- **Series gap false positives** — drafts intentionally numbered ahead of published. Mitigation: only validate contiguity within published posts; drafts excluded.
- **Audit becomes ritual without action** — same risk as ADR-001's app-audit. Mitigation: same abort condition — retire if two consecutive runs produce zero actionable findings.

## Implementation Notes

- `content-audit` SKILL.md location: `.agents/skills/content-audit/SKILL.md`, symlinked at `.claude/skills/content-audit`.
- Slash-command alias: `/content-audit` (replaces planned `/app-audit`).
- Per-run report location: `docs/_reports/content-audit-YYYY-MM-DD.md`.
- Committed summary location: `docs/audits/SUMMARY.md` (shared with future audit skills if any).
- New rule file: `.agents/rules/audit.md` describes the audit category and references content-audit (renamed from app-audit references).
- `app/lib/site-model.server.ts` exports both `routes` (for e2e-coverage) and `posts` (for content-audit).
- No changes required to `e2e-coverage` scope, fixture contract, PGLite setup, or CI gating from ADR-001.
- Drop from earlier plans:
  - `.agents/rules/audit.md` references to "console-error / network-fail / broken-link / slow-response / hydration-mismatch / a11y-violation" categories — replaced with content-audit categories: frontmatter-invalid, translation-gap, broken-link, missing-alt-text, series-gap.
  - "Two passes per run: anonymous session + admin session" — removed (no browser session).
  - a11y delegation to `a11y-testing` skill from audit — removed (different surface).
