# ADR-003: PostFrontmatter Canonical Type Uses String publishedAt

**Date:** 2026-05-06
**Status:** Accepted

## Decision

`app/types/content.ts` exports a single `PostFrontmatter` type with `publishedAt?: string`
(ISO 8601 string). The `indexer.ts` duplicate `interface PostFrontmatter` is removed;
`parseFrontmatterBlock` retains its internal Date handling via an inlined return type
rather than importing the shared type.

## Context

Two files defined `PostFrontmatter` with incompatible `publishedAt` types:
- `mdx.server.ts`: `publishedAt?: string` (gray-matter Date converted to ISO string for display)
- `indexer.ts`: `publishedAt?: Date` (raw Date object for Drizzle DB insertion)

A single shared type was preferred to eliminate duplication, but the types serve
different pipeline stages.

## Alternatives Considered

- **Two types (`PostFrontmatter` + `PostFrontmatterRaw`):** Explicit distinction but
  doubles the type surface. Over-engineering for a single-maintainer codebase.
- **Keep indexer.ts local type, move only mdx.server type:** Minimal disruption but
  leaves the duplication unresolved conceptually. The shared import adds no value
  if indexer can't use the same shape.

## Consequences

- `app/lib/mdx/parser.server.ts` imports and returns `PostFrontmatter` from `app/types/content.ts`
- `indexer.ts` removes `interface PostFrontmatter`; `parseFrontmatterBlock` uses an inlined
  return type `{ title: string; description?: string; publishedAt?: Date; slug?: string }`
  to preserve Date semantics for Drizzle inserts — no coercion needed
- Future callers that need the display-ready type import from `app/types/content.ts`
- Future callers that need DB-ready dates handle conversion locally (or a helper is
  added when the pattern repeats — YAGNI)
