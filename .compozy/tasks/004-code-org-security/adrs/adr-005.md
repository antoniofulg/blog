# ADR-005: MDX Directory Uses Direct Imports, No Barrel File

**Date:** 2026-05-06
**Status:** Accepted

## Decision

`app/lib/mdx/` has no `index.ts` barrel. Callers import directly from the specific
file they need: `#/lib/mdx/parser.server` or `#/lib/mdx/renderer.server`.

## Context

When `mdx.server.ts` is split into a directory with two files, a barrel `index.ts`
was considered to preserve the existing single-import ergonomic.

## Alternatives Considered

- **Barrel `index.ts`:** Single import path `#/lib/mdx` re-exports both parser and
  renderer. Rejected because TanStack Start's Vite-based bundler may not tree-shake
  barrel re-exports correctly in all configurations, risking `renderer.server` (which
  imports Shiki and @mdx-js/mdx) being bundled into client chunks that only need the parser.

## Consequences

- Two callers of `renderMdx` (`$slug.tsx`, `preview.$slug.tsx`) update imports to
  `#/lib/mdx/renderer.server` — both already use dynamic runtime imports inside
  `createServerFn`, so the path change is straightforward
- Import path is slightly more verbose but unambiguous about which concern is used
- Shiki + MDX compiler remain server-only — no accidental client bundle pollution
