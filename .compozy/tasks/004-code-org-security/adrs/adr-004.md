# ADR-004: requireSession Uses Throw-Only Pattern (Promise<void>)

**Date:** 2026-05-06
**Status:** Accepted

## Decision

`app/lib/session.ts` exports `requireSession(): Promise<void>`. The function throws
`new Response("Unauthorized", { status: 401 })` if no session exists; otherwise
returns void. Callers use it as a guard at the top of server function handlers.

## Context

The existing inline `requireSession` in `admin/index.tsx` and `admin/preview.$slug.tsx`
both return void. Two patterns were evaluated for the extracted shared utility.

## Alternatives Considered

- **Return `AuthUser`:** Caller gets `const user = await requireSession()`. Useful
  if server fns need `user.id` for scoped queries or audit logging. Rejected because:
  (1) no user-scoped queries exist or are planned for this single-admin blog;
  (2) adds `AuthUser` coupling at every call site; (3) callers that don't need the
  user object receive an ignored return value.

## Consequences

- Zero behavior change from current inline copies — drop-in extraction
- If a future server fn needs `user.id`, it must call `auth.api.getSession()` separately
  after `requireSession()` — accepted cost given current requirements
- If user-scoped features are added, this ADR should be revisited and `requireSession`
  updated to return `AuthUser` (all call sites need updating at that point)
