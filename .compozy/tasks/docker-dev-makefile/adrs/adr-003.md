# ADR-003: Multi-Stage Dockerfile with Named Build Targets

## Status

Accepted

## Date

2026-05-05

## Context

The blog uses TanStack Start with a Nitro `bun` preset. `vite build` outputs a self-contained server bundle to `.output/`. The dev containerized workflow (`make dev-docker`) needs a running Bun environment with source files mounted and `bun dev` as the entry point — not the production bundle. A single-stage Dockerfile cannot serve both purposes.

## Decision

Three-stage Dockerfile with explicit named targets:

1. **`dev` stage** — `oven/bun:1`, installs all deps, no COPY of source, CMD `bun dev`. Used by docker-compose app service for `make dev-docker`.
2. **`builder` stage** — extends `dev`, COPYs all source, runs `bun run build`. Intermediate stage only.
3. **`runner` stage** — `oven/bun:1-alpine`, copies only `.output/` from builder. Self-contained production image, CMD `bun .output/server/index.mjs`.

`make build` targets the `runner` stage (default). docker-compose `app` service targets the `dev` stage.

## Alternatives Considered

### Alternative 1: Two-stage (builder + runner only)

- **Description**: Single builder stage, single runner. No named dev target.
- **Pros**: Simpler Dockerfile; no multi-target complexity.
- **Cons**: `make dev-docker` would need a separate `Dockerfile.dev` or docker-compose override file; two files to maintain.
- **Why rejected**: Three stages in one Dockerfile is less complexity than two Dockerfiles.

### Alternative 2: docker-compose override file for dev

- **Description**: `docker-compose.yml` + `docker-compose.dev.yml` override. `make dev-docker` merges both.
- **Pros**: Clean separation of concerns.
- **Cons**: Two compose files; `make dev-docker` becomes `docker compose -f docker-compose.yml -f docker-compose.dev.yml watch`; harder to document.
- **Why rejected**: Single Dockerfile with named stages is simpler to understand for new contributors.

### Alternative 3: Distroless runner image

- **Description**: Use `gcr.io/distroless/base-debian12` as runner base.
- **Pros**: Smaller attack surface; no shell in production.
- **Cons**: Bun runtime must be copied from builder stage (adds complexity); no `make shell` in prod; Bun + distroless is poorly documented.
- **Why rejected**: `oven/bun:1-alpine` is the documented production pattern; distroless adds complexity without a meaningful security gain for a personal blog.

## Consequences

### Positive

- Single Dockerfile for all use cases
- `make build` and `make dev-docker` share the same base layer cache
- Runner image is minimal: `.output/` + Bun runtime only (Nitro bundles all deps)
- No production source code or devDependencies in the runner image

### Negative

- Three-stage Dockerfile is less obvious than a two-stage at first read
- `oven/bun:1` pinned to major version — minor/patch updates are automatic (acceptable risk)

### Risks

- Nitro bundle may reference external modules not inlined — mitigated by verifying `make preview` runs successfully before `make deploy`
- `.output/server/index.mjs` path is TanStack Start / Nitro convention — must be verified during implementation; update CMD if path differs

## Implementation Notes

```dockerfile
FROM oven/bun:1 AS dev
WORKDIR /app
COPY package.json bun.lock ./
RUN bun install --frozen-lockfile
CMD ["bun", "dev", "--port", "3000"]

FROM dev AS builder
COPY . .
RUN bun run build

FROM oven/bun:1-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production
COPY --from=builder /app/.output ./.output
EXPOSE 3000
CMD ["bun", ".output/server/index.mjs"]
```
