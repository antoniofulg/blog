# ADR-001: Containerized Dev as Opt-In Target, Not Default

## Status

Accepted

## Date

2026-05-04

## Context

The blog project (TanStack Start + Bun + Postgres) aims to add full Docker containerization and a Makefile to improve contributor onboarding. The key design decision is whether `make dev` should run the app natively (Bun on host + Postgres in Docker, current working state) or inside a container (full Docker compose stack).

Bun has a documented file-event propagation issue on macOS when running inside Docker containers (oven-sh/bun#9300). This affects hot reload reliability, which is central to the dev loop. The issue may require polling fallbacks or `docker compose watch` sync mode, both of which add complexity and degrade feedback speed.

## Decision

`make dev` defaults to native Bun dev server with Postgres via Docker Compose — matching the current working state but wrapped as a Makefile target.

`make dev-docker` is an explicit opt-in target for fully containerized dev, using `docker compose watch` sync mode with documented platform caveats (macOS hot-reload gotchas).

## Alternatives Considered

### Alternative 1: Containerized dev as default (`make dev`)

- **Description**: `make dev` = `docker compose up` with full app + DB containers
- **Pros**: True "only need Docker Desktop" onboarding; CI environment parity
- **Cons**: Exposes all contributors to the Bun/macOS hot-reload reliability issue; cold-start latency; complex volume strategy required
- **Why rejected**: Puts the known-risky path at the front door; a broken welcome mat is worse than no welcome mat

### Alternative 2: No Docker for dev at all

- **Description**: Makefile wraps existing `bun` scripts only; contributors install Bun + Postgres locally
- **Pros**: Zero Docker complexity; native speed guaranteed
- **Cons**: Contributors need Bun + Postgres installed; no environment parity with CI; higher per-contributor setup friction
- **Why rejected**: Defeats the stated goal of reducing contributor onboarding friction

## Consequences

### Positive

- Native dev loop remains fast and reliable (hot reload confirmed working)
- Contributors can opt into full container experience when they want CI parity
- Clear upgrade path to default-container once Bun macOS issue is resolved
- `make dev` promise is honest and delivers reliable hot reload from day one

### Negative

- Two dev targets to maintain (`dev` and `dev-docker`)
- Contributors running `make dev-docker` may hit platform-specific issues without guidance
- "Only Docker Desktop needed" onboarding claim requires `make dev-docker`, not `make dev`

### Risks

- `make dev-docker` hot reload may be unreliable on macOS without polling or compose watch — mitigated by documenting the `CHOKIDAR_USEPOLLING` flag and `compose watch` alternative in the Makefile help text
- Contributors may conflate the two targets — mitigated by a clear `make help` description

## Implementation Notes

- `make dev` → `docker compose up db -d && bun dev`
- `make dev-docker` → `docker compose up` (full stack with `develop: watch:` blocks in compose)
- The compose `watch:` sync block handles file sync without relying on native inotify events
